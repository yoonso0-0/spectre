      27             : <sect1 id="versioning_and_releases_1autotoc_md175">
      28             : <title>Version format</title>
      29             : <para>We employ a date-based versioning scheme with the format <computeroutput>YYYY.0M.0D[.TWEAK]</computeroutput>, where <computeroutput>YYYY</computeroutput> is the year, <computeroutput>0M</computeroutput> the zero-padded month and <computeroutput>0D</computeroutput> the zero-padded day of the month when the version is released. If additional versions are released on a single day, the <computeroutput>TWEAK</computeroutput> component enumerates them. For example, the first version released on May 4th, 2021 carries the name <computeroutput>2021.05.04</computeroutput>. Additional versions released on May 4th, 2021 would carry the names <computeroutput>2021.05.04.1</computeroutput>, <computeroutput>2021.05.04.2</computeroutput>, etc.</para>
      30             : <para>The reason we use a date-based versioning scheme over, for example, semantic versioning is to avoid implying a notion of compatibility between releases (see section below).</para>
      31             : </sect1>
      32             : <sect1 id="versioning_and_releases_1autotoc_md176">
      33             : <title>Guarantees and non-guarantees of releases</title>
      34             : <para>Each release is <bold>guaranteed</bold> to pass all automated tests laid out in the <ref refid="travis_guide" kindref="compound">guide on automated tests</ref>.</para>
      35             : <para>We provide <bold>no guarantee</bold> that releases satisfy any notion of &quot;compatibility&quot; with each other. In particular, the interface of any C++ library, Python module or executable may change between releases. When writing code for SpECTRE we recommend regularly rebasing on the latest <computeroutput>develop</computeroutput> branch of the <ulink url="https://github.com/sxs-collaboration/spectre"><computeroutput>sxs-collaboration/spectre</computeroutput></ulink> repository and aiming to contribute your code upstream by following our <ref refid="contributing_to_spectre" kindref="compound">contributing guide</ref> in a timely manner. That said, we make an effort to retain compatibility between releases for the following aspects of the code:</para>
      36             : <para><itemizedlist>
      37             : <listitem><para>Simulation data: We try to ensure that data produced by a release remains compatible with future releases, in the sense that the data can be read, processed or converted to newer formats.</para>
      38             : </listitem><listitem><para>Python bindings: We try to offer deprecation warnings when changing the Python interfaces so developers of external packages that use the Python bindings have time to update their packages.</para>
      39             : </listitem></itemizedlist>
      40             : </para>
      41             : <para>We may establish guarantees for specific interfaces related to these aspects. These guarantees will be defined in the documentation of the respective interface.</para>
      42             : <para>We explicitly provide <bold>no guarantee</bold> that input files remain compatible with executables built on different releases. Instead, we try to make input files as explicit as possible by not supporting default values, so simulations don&apos;t continue with subtly changed parameters part way through. We try to reflect changes that would affect running simulations in the input files, so they can (and often must) be reviewed before the simulation continues. Please note that low-level changes or bugs may always alter results, so be advised to stick with a particular compiled executable if a high level of reproducibility is crucial for your project.</para>
      43             : </sect1>
