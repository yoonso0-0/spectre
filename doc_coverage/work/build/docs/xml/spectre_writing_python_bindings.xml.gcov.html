      10             : <title>CMake and Directory Layout</title>
      11             : <para>To allow users to analyze output from simulations and take advantage of SpECTRE&apos;s data structures and functions in python, bindings must sometimes be written. SpECTRE uses <ulink url="https://pybind11.readthedocs.io/">pybind11</ulink> to aid with generating the bindings. The C++ code for the bindings should generally go in a <computeroutput>Python</computeroutput> subdirectory. For example, the bindings for the DataStructures library would go in <computeroutput>src/DataStructures/Python/</computeroutput>. SpECTRE provides the <computeroutput>spectre_python_add_module</computeroutput> CMake function to make adding a new python module, be it with or without bindings, easy. The python bindings are built only if <computeroutput>-D BUILD_PYTHON_BINDINGS=ON</computeroutput> is passed when invoking cmake. You can specify the Python version, interpreter and libraries used for compiling and testing the bindings by setting the <computeroutput>-D Python_EXECUTABLE</computeroutput> to an absolute path such as <computeroutput>/usr/bin/python3</computeroutput>.</para>
      12             : <para>The function <computeroutput>spectre_python_add_module</computeroutput> takes as its first argument the module, in our case <computeroutput>DataStructures</computeroutput>. Optionally, a list of <computeroutput>SOURCES</computeroutput> can be passed to the CMake function. If you specify <computeroutput>SOURCES</computeroutput>, you must also specify a <computeroutput>LIBRARY_NAME</computeroutput>. A good <computeroutput>LIBRARY_NAME</computeroutput> is the name of the C++ library for which bindings are being built prefixed with <computeroutput>Py</computeroutput>, e.g. <computeroutput>PyDataStructures</computeroutput>. If the Python module will only consist of Python files, then the <computeroutput>SOURCES</computeroutput> option should not be specified. Python files that should be part of the module can be passed with the keywords <computeroutput>PYTHON_FILES</computeroutput> or <computeroutput>PYTHON_EXECUTABLES</computeroutput>, where the latter exposes them in <computeroutput>bin/</computeroutput> in the SpECTRE build directory. For example, the arguments <computeroutput>PYTHON_FILES Hello.py PYTHON_EXECUTABLES HelloWorld.py</computeroutput> produce a Python module that contains both files, but <computeroutput>HelloWorld.py</computeroutput> can be run directly as <computeroutput>bin/HelloWorld</computeroutput> from the SpECTRE build directory. Finally, the <computeroutput>MODULE_PATH</computeroutput> named argument can be passed with a string that is the path to where the module should be. For example, <computeroutput>MODULE_PATH &quot;submodule0/submodule1/&quot;</computeroutput> would mean the module is accessed from python using <computeroutput>import spectre.submodule0.submodule1.MODULE_NAME</computeroutput>.</para>

      40             : <title>Writing Bindings</title>
      41             : <para>Once a python module has been added you can write the actual bindings. You should structure your bindings directory to reflect the structure of the library you&apos;re writing bindings for. For example, say we want bindings for <computeroutput><ref refid="classDataVector" kindref="compound">DataVector</ref></computeroutput> and <computeroutput><ref refid="classMatrix" kindref="compound">Matrix</ref></computeroutput> then we should have one source file for each class&apos;s bindings inside <computeroutput>src/DataStructures/Python</computeroutput>. The functions that generate the bindings should be in the <computeroutput>py_bindings</computeroutput> namespace and have a reasonable name such as <computeroutput>bind_datavector</computeroutput>. There should be a file named <computeroutput>Bindings.cpp</computeroutput> which calls all the <computeroutput>bind_*</computeroutput> functions. The <computeroutput>Bindings.cpp</computeroutput> file is quite simple and should <computeroutput>include &lt;pybind11/pybind11.h&gt;</computeroutput>, forward declare the <computeroutput>bind_*</computeroutput> functions, and then have a <computeroutput>PYBIND11_MODULE</computeroutput> function. For example,</para>

      55             : <para>The <computeroutput><ref refid="classDataVector" kindref="compound">DataVector</ref></computeroutput> bindings serve as an example with code comments on how to write bindings for a class. SpECTRE currently aims to support both Python 2.7 and Python 3 and as such all bindings must support both.</para>

      61             : <title>Testing Python Bindings and Code</title>
      62             : <para>All the python bindings must be tested. SpECTRE uses the <ulink url="https://docs.python.org/3/library/unittest.html">unittest</ulink> framework provided as part of python. To register a test file with CMake use the SpECTRE-provided function <computeroutput>spectre_add_python_test</computeroutput> passing as the first argument the test name (e.g. <computeroutput>&quot;Unit.DataStructures.Python.DataVector&quot;</computeroutput>), the file as the second argument (e.g. <computeroutput>Test_DataVector.py</computeroutput>), and a semicolon separated list of labels as the last (e.g. <computeroutput>&quot;unit;datastructures;python&quot;</computeroutput>). All the test cases should be in a single class so that the python unit testing framework will run all test functions on a single invocation to avoid startup cost.</para>
