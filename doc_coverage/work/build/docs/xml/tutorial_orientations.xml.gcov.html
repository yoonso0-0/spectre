       9             : <sect2 id="tutorial_orientations_1autotoc_md189">
      10             : <title>Introduction</title>
      11             : <para>Each element in a domain has a set of internal directions which it uses for computations in its own local coordinate system. These are referred to as the logical directions <formula id="324">$\xi$</formula>, <formula id="325">$\eta$</formula>, and <formula id="285">$\zeta$</formula>, where <formula id="324">$\xi$</formula> is the first dimension, <formula id="325">$\eta$</formula> is the second dimension, and <formula id="285">$\zeta$</formula> is the third dimension. In a domain with multiple elements, the logical directions are not necessarily aligned on the interfaces between two elements, as shown in the figure below. As certain operations (e.g. fluxes, limiting) communicate information across the boundaries of adjacent elements, there needs to be a class that takes into account the relative orientations of elements which neighbor each other. This class is OrientationMap.</para>
      12             : </sect2>
      13             : <sect2 id="tutorial_orientations_1autotoc_md190">
      14             : <title>%OrientationMaps between %Blocks</title>
      15             : <para>Each <ref refid="classBlock" kindref="compound">Block</ref> in a <ref refid="classDomain" kindref="compound">Domain</ref> has a set of BlockNeighbors, which each hold an OrientationMap. In this scenario, the <ref refid="classBlock" kindref="compound">Block</ref> is referred to as the host, and the OrientationMap held by each <ref refid="classBlockNeighbor" kindref="compound">BlockNeighbor</ref> is referred to as &quot;the
      16             : orientation the BlockNeighbor has with respect to the host Block.&quot; This is a convention, so we give an example of constructing and assigning the correct OrientationMaps:</para>
      17             : <para><image type="html" name="twocubes.png">Two neighboring blocks.</image>
      18             : </para>
      19             : <para>In the image above, we see a domain decomposition into two Blocks, which have their logical axes rotated relative to one another. With the left block as the host <ref refid="classBlock" kindref="compound">Block</ref>, we see that it has a neighbor in the <formula id="2279">$+\xi$</formula> direction. The host <ref refid="classBlock" kindref="compound">Block</ref> holds a <computeroutput><ref refid="cpp/container/unordered_map" kindref="compound" external="/__w/spectre/spectre/docs/config/cppreference-doxygen-web.tag.xml">std::unordered_map</ref></computeroutput> from Directions to BlockNeighbors; the <ref refid="classBlockNeighbor" kindref="compound">BlockNeighbor</ref> itself holds an OrientationMap that determines the mapping from each logical direction in the host <ref refid="classBlock" kindref="compound">Block</ref> to that in the neighboring <ref refid="classBlock" kindref="compound">Block</ref>. That is, the OrientationMap takes as input local information (i.e. logical directions in the host&apos;s coordinate system) and returns neighbor information (i.e. logical directions in the neighbor&apos;s coordinate system). An OrientationMap is constructed by passing in the block neighbor directions that correspond to the <formula id="2279">$+\xi$</formula>, <formula id="2280">$+\eta$</formula>, <formula id="290">$+\zeta$</formula> directions in the host. In this case, these directions in the host map to the <formula id="290">$+\zeta$</formula>, <formula id="2279">$+\xi$</formula>, <formula id="2280">$+\eta$</formula> directions in the neighbor, respectively. This <ref refid="classBlockNeighbor" kindref="compound">BlockNeighbor</ref> thus holds the OrientationMap constructed with the list ( <formula id="290">$+\zeta$</formula>, <formula id="2279">$+\xi$</formula>, <formula id="2280">$+\eta$</formula>). With the right block as the host block, we see that it has a <ref refid="classBlockNeighbor" kindref="compound">BlockNeighbor</ref> in the <formula id="2281">$-\zeta$</formula> direction, and the OrientationMap held by this <ref refid="classBlockNeighbor" kindref="compound">BlockNeighbor</ref> is the one constructed with the array ( <formula id="2280">$+\eta$</formula>, <formula id="290">$+\zeta$</formula>, <formula id="2279">$+\xi$</formula>). For convenience, OrientationMap has a method <computeroutput>inverse_map</computeroutput> which returns the OrientationMap that takes as input neighbor information and returns local information.</para>
      20             : <para>OrientationMaps need to be provided for each <ref refid="classBlockNeighbor" kindref="compound">BlockNeighbor</ref> in each direction for each <ref refid="classBlock" kindref="compound">Block</ref>. This quickly becomes too large of a number to determine by hand as the number of Blocks and the number of dimensions increases. A remedy to this problem is the corner numbering scheme.</para>
      21             : </sect2>
      22             : <sect2 id="tutorial_orientations_1autotoc_md191">
      23             : <title>Encoding BlockNeighbor information using Corner Orderings and Numberings</title>
      24             : <para>The orientation of the <formula id="2282">${dim}$</formula> logical directions within each element determines an ordering of the <formula id="2283">$2^{dim}$</formula> vertices of that element. This is called the local corner numbering scheme (Local CNS) with respect to that element. We give the ordering of the local corners below for the case of a three-dimensional element:</para>
      25             : <para><image type="html" name="onecube_numbered.png">The local corner numbering.</image>
      26             : </para>
      27             : <para><programlisting><codeline><highlight class="normal">Corner<sp/>0<sp/>is<sp/>the<sp/>location<sp/>of<sp/>the<sp/>lower<sp/>xi,<sp/>lower<sp/>eta,<sp/>lower<sp/>zeta<sp/>corner.</highlight></codeline>
      28             : <codeline><highlight class="normal">Corner<sp/>1<sp/>is<sp/>the<sp/>location<sp/>of<sp/>the<sp/>upper<sp/>xi,<sp/>lower<sp/>eta,<sp/>lower<sp/>zeta<sp/>corner.</highlight></codeline>
      29             : <codeline><highlight class="normal">Corner<sp/>2<sp/>is<sp/>the<sp/>location<sp/>of<sp/>the<sp/>lower<sp/>xi,<sp/>upper<sp/>eta,<sp/>lower<sp/>zeta<sp/>corner.</highlight></codeline>
      30             : <codeline><highlight class="normal">Corner<sp/>3<sp/>is<sp/>the<sp/>location<sp/>of<sp/>the<sp/>upper<sp/>xi,<sp/>upper<sp/>eta,<sp/>lower<sp/>zeta<sp/>corner.</highlight></codeline>
      31             : <codeline><highlight class="normal">Corner<sp/>4<sp/>is<sp/>the<sp/>location<sp/>of<sp/>the<sp/>lower<sp/>xi,<sp/>lower<sp/>eta,<sp/>upper<sp/>zeta<sp/>corner.</highlight></codeline>
      32             : <codeline><highlight class="normal">Corner<sp/>5<sp/>is<sp/>the<sp/>location<sp/>of<sp/>the<sp/>upper<sp/>xi,<sp/>lower<sp/>eta,<sp/>upper<sp/>zeta<sp/>corner.</highlight></codeline>
      33             : <codeline><highlight class="normal">Corner<sp/>6<sp/>is<sp/>the<sp/>location<sp/>of<sp/>the<sp/>lower<sp/>xi,<sp/>upper<sp/>eta,<sp/>upper<sp/>zeta<sp/>corner.</highlight></codeline>
      34             : <codeline><highlight class="normal">Corner<sp/>7<sp/>is<sp/>the<sp/>location<sp/>of<sp/>the<sp/>upper<sp/>xi,<sp/>upper<sp/>eta,<sp/>upper<sp/>zeta<sp/>corner.</highlight></codeline>
      35             : </programlisting></para>
      36             : <para>What remains is to endow the domain decomposition with a global corner numbering (Global CNS). We give an example below:</para>
      37             : <para><image type="html" name="twocubes_numbered.png">A global corner numbering.</image>
      38             : </para>
      39             : <para>In the image above, we see that each vertex of the two-block domain has been assigned a number. Although each block has eight corners, four are shared among them, so there are only twelve unique corners in this domain. Any numbering may be used in the global corner numbering, so long as the each distinct corner is given a single distinct corner number.</para>
      40             : <para><simplesect kind="note"><para>This Global CNS assumes that there is no additional identifying of faces with one another for periodic boundary conditions. That is, each element must have <formula id="2283">$2^{dim}$</formula> distinct corner numbers. If you wish to additionally identify faces of the same block with each other, that must be done in an additional step. This step is explained in the &quot;Setting Periodic Boundary
      41             : Conditions&quot; section.</para>
      42             : </simplesect>
      43             : </para>

      45             : <sect2 id="tutorial_orientations_1autotoc_md192">
      46             : <title>The Ordered Subset of the Global CNS (Subset CNS):</title>
      47             : <para>With the Global CNS in hand, each <ref refid="classBlock" kindref="compound">Block</ref> inherits an ordered subset of Global CNS. The ordering in this set is determined by the ordering of the Local CNS, and the elements of the set determined by how one assigned the Global CNS to the <ref refid="classDomain" kindref="compound">Domain</ref>. For the image above, the Subset CNS corresponding to the left block is {0, 1, 3, 4, 6, 7, 9, 10}, while the Subset CNS corresponding to the right block is {1, 4, 7, 10, 2, 5, 8, 11}. This ordering of the Subset CNS encodes the relative orientations between each <ref refid="classBlock" kindref="compound">Block</ref>. Subset CNSs need to be provided for each <ref refid="classBlock" kindref="compound">Block</ref> in a <ref refid="classDomain" kindref="compound">Domain</ref>. It turns out that for very regular domains, (i.e. spherical or rectilinear) we can generate the appropriate Subset CNSs. As this is a conceptual tutorial, how to construct these domains in SpECTRE is described in the <ref refid="tutorial_domain_creation" kindref="compound">Domain Creation</ref> tutorial.</para>
      48             : </sect2>
      49             : <sect2 id="tutorial_orientations_1autotoc_md193">
      50             : <title>Explanation of the Algorithms in DomainHelpers:</title>
      51             : <para>For illustrative purposes, we will use the following <ref refid="classDomain" kindref="compound">Domain</ref> composed of two Blocks as described above as an example. Because there are 12 corners in this <ref refid="classDomain" kindref="compound">Domain</ref>, we will arbitrarily assign a unique id to each corner. Knowing the orientation of the logical axes within a block, we construct a Subset CNS for each <ref refid="classBlock" kindref="compound">Block</ref>.<linebreak/>
      52             :  Here is one possible result, given some relative orientation between the blocks:</para>
      53             : <para><programlisting><codeline><highlight class="normal">Block1:<sp/>{0,<sp/>1,<sp/>3,<sp/>4,<sp/>6,<sp/>7,<sp/>9,<sp/>10}</highlight></codeline>
      54             : <codeline><highlight class="normal">Block2:<sp/>{1,<sp/>4,<sp/>7,<sp/>10,<sp/>2,<sp/>5,<sp/>8,<sp/>11}</highlight></codeline>
      55             : </programlisting></para>
      56             : <para>The values of the ids only serve to identify which corners are unique and which are shared. This is determined by the Global CNS. The order of the ids in the list is determined by the Local CNS. We take advantage of the fact that the array index of the global corner id is the number of the corner in the local CNS.</para>
      57             : <para>The algorithm begins by determining the shared corners between the Blocks:</para>
      58             : <para><programlisting><codeline><highlight class="normal">result:<sp/>{1,<sp/>4,<sp/>7,<sp/>10}</highlight></codeline>
      59             : </programlisting></para>
      60             : <para>The next step is to determine the local ids of these shared global ids:</para>
      61             : <para><programlisting><codeline><highlight class="normal">Block1<sp/>result:<sp/>{1,3,5,7}</highlight></codeline>
      62             : <codeline><highlight class="normal">Block2<sp/>result:<sp/>{0,1,2,3}</highlight></codeline>
      63             : </programlisting></para>
      64             : <para>The next step is to convert these ids into their binary representation. For reference, we give the binary representation for each number 0-7:</para>
      65             : <para><programlisting><codeline><highlight class="normal">Corner<sp/>0:<sp/>000</highlight></codeline>
      66             : <codeline><highlight class="normal">Corner<sp/>1:<sp/>001</highlight></codeline>
      67             : <codeline><highlight class="normal">Corner<sp/>2:<sp/>010</highlight></codeline>
      68             : <codeline><highlight class="normal">Corner<sp/>3:<sp/>011</highlight></codeline>
      69             : <codeline><highlight class="normal">Corner<sp/>4:<sp/>100</highlight></codeline>
      70             : <codeline><highlight class="normal">Corner<sp/>5:<sp/>101</highlight></codeline>
      71             : <codeline><highlight class="normal">Corner<sp/>6:<sp/>110</highlight></codeline>
      72             : <codeline><highlight class="normal">Corner<sp/>7:<sp/>111</highlight></codeline>
      73             : </programlisting></para>
      74             : <para>Here 0 and 1 indicate lower and upper in the corresponding axis (zeta, eta, xi), respectively, and the ordering has been reversed so that the rightmost column corresponds to the xi position and the leftmost column to the zeta position. Returning to the example at hand, we have:</para>
      75             : <para><programlisting><codeline><highlight class="normal">Block1<sp/>result:<sp/>{001,011,101,111}</highlight></codeline>
      76             : <codeline><highlight class="normal">Block2<sp/>result:<sp/>{000,001,010,011}</highlight></codeline>
      77             : </programlisting></para>
      78             : <para>Note that we can now read off the shared face relative to each <ref refid="classBlock" kindref="compound">Block</ref> easily:</para>
      79             : <para><programlisting><codeline><highlight class="normal">Block1<sp/>result:<sp/>Upper<sp/>xi<sp/>(All<sp/>binary<sp/>Block1<sp/>ids<sp/>have<sp/>a<sp/>1<sp/>in<sp/>the<sp/>third<sp/>position)</highlight></codeline>
      80             : <codeline><highlight class="normal">Block2<sp/>result:<sp/>Lower<sp/>zeta<sp/>(All<sp/>binary<sp/>Block2<sp/>ids<sp/>have<sp/>a<sp/>0<sp/>in<sp/>the<sp/>first<sp/>position)</highlight></codeline>
      81             : </programlisting></para>
      82             : <para>Now we know that <computeroutput><ref refid="classDirection" kindref="compound">Direction</ref>&lt;3&gt;::upper_xi()</computeroutput> in Block1 corresponds to <computeroutput><ref refid="classDirection" kindref="compound">Direction</ref>&lt;3&gt;::lower_zeta().opposite()</computeroutput> in Block2.</para>
      83             : <para>The use of <computeroutput>.opposite()</computeroutput> is a result of the Directions to a <ref refid="classBlock" kindref="compound">Block</ref> face being anti-parallel because each <ref refid="classBlock" kindref="compound">Block</ref> lies on the opposite side of the shared face. <linebreak/>
      84             : </para>
      85             : <para>The remaining two correspondences are given by the alignment of the shared face. It is useful to know the following information:<linebreak/>
      86             :  In the Local CNS, if an edge lies along the xi direction, if one takes the two corners making up that edge and takes the difference of their ids, one always gets the result <formula id="2284">$ \pm 1$</formula>. Similarly, if the edge lies in the eta direction, the result will be <formula id="2285">$ \pm 2$</formula>. Finally, if the edge lies in the zeta direction, the result will be <formula id="2286">$ \pm 4$</formula>. We use this information to determine the alignment of the shared face:</para>
      87             : <para><programlisting><codeline><highlight class="normal">Block1:<sp/>3-1=2<sp/><sp/>=&gt;<sp/>This<sp/>edge<sp/>is<sp/>in<sp/>the<sp/>+eta<sp/>direction.</highlight></codeline>
      88             : <codeline><highlight class="normal">Block2:<sp/>1-0=1<sp/>=&gt;<sp/>This<sp/>edge<sp/>is<sp/>in<sp/>the<sp/>+xi<sp/>direction.</highlight></codeline>
      89             : <codeline><highlight class="normal">Then,<sp/>+eta<sp/>in<sp/>Block1<sp/>corresponds<sp/>to<sp/>+xi<sp/>in<sp/>Block2.</highlight></codeline>
      90             : <codeline></codeline>
      91             : <codeline><highlight class="normal">Block1:<sp/>5-1=4<sp/>=&gt;<sp/>This<sp/>edge<sp/>is<sp/>in<sp/>the<sp/>+zeta<sp/>direction.</highlight></codeline>
      92             : <codeline><highlight class="normal">Block2:<sp/>2-0=2<sp/>=&gt;<sp/>This<sp/>edge<sp/>is<sp/>in<sp/>the<sp/>+eta<sp/>direction.</highlight></codeline>
      93             : <codeline><highlight class="normal">Then,<sp/>+zeta<sp/>in<sp/>Block1<sp/>corresponds<sp/>to<sp/>+eta<sp/>in<sp/>Block2.</highlight></codeline>
      94             : </programlisting></para>
      95             : <para>The corresponding directions in each <ref refid="classBlock" kindref="compound">Block</ref> have now been deduced.</para>
      96             : <para>To confirm, we can use the other ids as well and arrive at the same result:<linebreak/>
      97             : </para>
      98             : <para><programlisting><codeline><highlight class="normal">Block1:<sp/>7-5=2<sp/><sp/>=&gt;<sp/>+eta</highlight></codeline>
      99             : <codeline><highlight class="normal">Block2:<sp/>3-2=1<sp/>=&gt;<sp/>+xi</highlight></codeline>
     100             : <codeline></codeline>
     101             : <codeline><highlight class="normal">Block1:<sp/>7-3=4<sp/><sp/>=&gt;<sp/>+zeta</highlight></codeline>
     102             : <codeline><highlight class="normal">Block2:<sp/>3-1=2<sp/><sp/>=&gt;<sp/>+eta</highlight></codeline>
     103             : </programlisting></para>

     106             : <title>Setting Periodic Boundary Conditions</title>
     107             : <para>It is also possible to identify faces of a <ref refid="classBlock" kindref="compound">Block</ref> using the subset CNS. For example, to identify the lower zeta face with the upper zeta face of a <ref refid="classBlock" kindref="compound">Block</ref> where the corners are labeled <computeroutput>{3,0,4,1,9,6,10,7}</computeroutput>, one may supply the lists <computeroutput>{3,0,4,1}</computeroutput> and <computeroutput>{9,6,10,7}</computeroutput> to the <computeroutput>set_identified_boundaries</computeroutput> function. <simplesect kind="note"><para>The <computeroutput>set_identified_boundaries</computeroutput> function is sensitive to the order of the corners in the lists supplied as arguments. This is because the function identifies corners and edges with each other as opposed to simply faces. This allows the user to specify more peculiar boundary conditions. For example, using <computeroutput>{3,0,4,1}</computeroutput> and <computeroutput>{6,7,9,10}</computeroutput> to set the periodic boundaries will identify the lower zeta face with the upper zeta face, but after a rotation of a quarter-turn.</para>
