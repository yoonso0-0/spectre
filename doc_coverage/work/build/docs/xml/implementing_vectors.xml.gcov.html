      79             : <sect1 id="implementing_vectors_1general_structure">
      80             : <title>Overview of SpECTRE Vectors</title>
      81             : <para>In SpECTRE, sets of contiguous or related data are stored in specializations of vector data types. The canonical implementation of this is the <computeroutput><ref refid="classDataVector" kindref="compound">DataVector</ref></computeroutput>, which is used for storage of a contiguous sequence of doubles which support a wide variety of mathematical operations and represent data on a grid used during an evolution or elliptic solve. However, we support the ability to easily generate similar vector types which can hold data of a different type (e.g. <computeroutput><ref refid="cpp/numeric/complex" kindref="compound" external="/__w/spectre/spectre/docs/config/cppreference-doxygen-web.tag.xml">std::complex</ref>&lt;double&gt;</computeroutput>), or support a different set of mathematical operations. SpECTRE vector classes are derived from the class template <computeroutput><ref refid="classVectorImpl" kindref="compound">VectorImpl</ref></computeroutput>. The remainder of this brief guide gives a description of the tools for defining additional vector types.</para>
      82             : <para>For reference, all functions described here can also be found in brief in the Doxygen documentation for <ref refid="VectorImpl_8hpp_source" kindref="compound">VectorImpl.hpp</ref>, and a simple reference implementation can be found in <ref refid="DataVector_8hpp_source" kindref="compound">DataVector.hpp</ref> and DataVector.cpp.</para>
      83             : </sect1>
      84             : <sect1 id="implementing_vectors_1class_definition">
      85             : <title>The class definition</title>
      86             : <para>SpECTRE vector types inherit from vector types implemented in the high-performance arithmetic library <ulink url="https://bitbucket.org/blaze-lib/blaze">Blaze</ulink>. Using inheritance, SpECTRE vectors gracefully make use of the math functions defined for the Blaze types, but can be customized for the specific needs in SpECTRE computations.</para>
      87             : <para>The pair of template parameters for <computeroutput><ref refid="classVectorImpl" kindref="compound">VectorImpl</ref></computeroutput> are the type of the stored data (e.g. <computeroutput>double</computeroutput> for <computeroutput><ref refid="classDataVector" kindref="compound">DataVector</ref></computeroutput>), and the result type for mathematical operations. The result type is used by Blaze to ensure that only compatible vector types are used together in mathematical expressions. For example, a vector representing <computeroutput>double</computeroutput> data on a grid (<computeroutput><ref refid="classDataVector" kindref="compound">DataVector</ref></computeroutput>) cannot be added to a vector representing spectral coefficients (<computeroutput><ref refid="classModalVector" kindref="compound">ModalVector</ref></computeroutput>). This avoids subtle bugs that arise when vector types are unintentionally mixed. In nearly all cases the result type will be the vector type that is being defined, so, for instance, <computeroutput><ref refid="classDataVector" kindref="compound">DataVector</ref></computeroutput> is a derived class of <computeroutput><ref refid="classVectorImpl" kindref="compound">VectorImpl</ref>&lt;double, <ref refid="classDataVector" kindref="compound">DataVector</ref>&gt;</computeroutput>. This template pattern is known as the <ulink url="https://en.wikipedia.org/wiki/Curiously_recurring_template_pattern">&quot;Curiously Recurring Template Pattern&quot;</ulink> (CRTP).</para>
      88             : <para>The class template <computeroutput><ref refid="classVectorImpl" kindref="compound">VectorImpl</ref></computeroutput> defines various constructors, assignment operators, and iterator generation members. Most of these are inherited from Blaze types, but in addition, the methods <computeroutput>set_data_ref</computeroutput>, and <computeroutput>pup</computeroutput> are defined for use in SpECTRE. All except for the assignment operators and constructors will be implicitly inherited from <computeroutput><ref refid="classVectorImpl" kindref="compound">VectorImpl</ref></computeroutput>. The assignment and constructors may be inherited calling the following alias code in the vector class definition:</para>
      89             : <para><programlisting><codeline><highlight class="normal">using<sp/>VectorImpl&lt;T,VectorType&gt;::operator=;</highlight></codeline>
      90             : <codeline><highlight class="normal">using<sp/>VectorImpl&lt;T,VectorType&gt;::VectorImpl;</highlight></codeline>
      91             : </programlisting></para>
      92             : <para>Only the mathematical operations supported on the base Blaze types are supported by default. Those operations are determined by the storage type <computeroutput>T</computeroutput> and by the Blaze library. See <ulink url="https://bitbucket.org/blaze-lib/blaze/wiki/Vector%20Operations">blaze-wiki/Vector_Operations</ulink>.</para>
      93             : <para>Other math operations may be defined either in the class definition or outside. For ease in defining some of these math operations, <computeroutput><ref refid="PointerVector_8hpp" kindref="compound">PointerVector.hpp</ref></computeroutput> defines the macro <computeroutput>MAKE_EXPRESSION_MATH_ASSIGN(OP, TYPE)</computeroutput>, which can be used to define math assignment operations such as <computeroutput>+=</computeroutput> provided the underlying operation (e.g. <computeroutput>+</computeroutput>) is defined on the Blaze type.</para>
      94             : </sect1>

     169             : <sect1 id="implementing_vectors_1Vector_MakeWithValueImpl">
     170             : <title>MakeWithValueImpl</title>
     171             : <para>SpECTRE offers the convenience function <computeroutput>make_with_value</computeroutput> for various types. The typical behavior for a SpECTRE vector type is to create a new vector type of the same type and length initialized with the value provided as the second argument in all entries. This behavior may be created by placing the macro <computeroutput><ref refid="group__DataStructuresGroup_1gabe9b48242b12719bc435709531049d47" kindref="member">MAKE_WITH_VALUE_IMPL_DEFINITION_FOR(VECTOR_TYPE)</ref></computeroutput> in the .hpp file. Any other specializations of <computeroutput>MakeWithValueImpl</computeroutput> will need to be written manually.</para>

     173             : <sect1 id="implementing_vectors_1Vector_tensor_and_variables">
     174             : <title>Interoperability with other data types</title>
     175             : <para>When additional vector types are added, small changes are necessary if they are to be used as the base container type either for <computeroutput>Tensor</computeroutput>s or for <computeroutput>Variables</computeroutput> (a <computeroutput>Variables</computeroutput> contains <computeroutput>Tensor</computeroutput>s), which contain some vector type.</para>

     254             : <sect1 id="implementing_vectors_1Vector_storage">
     255             : <title>Vector storage nuts and bolts</title>
     256             : <para>Internally, all vector classes inherit from the templated <computeroutput><ref refid="classVectorImpl" kindref="compound">VectorImpl</ref></computeroutput>, which inherits from <computeroutput><ref refid="structPointerVector" kindref="compound">PointerVector</ref></computeroutput>, which inherits from a <computeroutput>blaze::DenseVector</computeroutput>. Most of the mathematical operations are supported through the Blaze inheritance, which ensures that the math operations execute the optimized forms in Blaze.</para>
     257             : <para>SpECTRE vectors can be either &quot;owning&quot; or &quot;non-owning&quot;. If a vector is owning, it allocates and controls the data it has access to, and is responsible for eventually freeing that data when the vector goes out of scope. If the vector is non-owning, it acts as a (possibly complete) &quot;view&quot; of otherwise allocated memory. Non-owning vectors do not manage memory, nor can they change size. The two cases of data ownership cause the underlying data to be handled fairly differently, so we will discuss each in turn.</para>
     258             : <para>In both cases of ownership, the base <computeroutput><ref refid="structPointerVector" kindref="compound">PointerVector</ref></computeroutput> contains a pointer to the allocated contiguous block of memory via a raw pointer <computeroutput>v_</computeroutput> (accessible via <computeroutput>PointerVector.data()</computeroutput> and the extent of that memory <computeroutput>size_</computeroutput> (accessible via <computeroutput>PointerVector.size()</computeroutput>). The raw pointer in <computeroutput><ref refid="structPointerVector" kindref="compound">PointerVector</ref></computeroutput> then gives access to the memory block to the base Blaze types, which perform the work of the actual math operations. <computeroutput><ref refid="structPointerVector" kindref="compound">PointerVector</ref></computeroutput> is closely patterned off of Blaze internal functionality (<computeroutput>CustomVector</computeroutput>), and we stress that direct alteration of <computeroutput><ref refid="structPointerVector" kindref="compound">PointerVector</ref></computeroutput> should be avoided unless completely necessary. The discussion in this section is intended to provide a better understanding of how the interface is used for the SpECTRE vector types so that future customization of derived classes of <computeroutput><ref refid="classVectorImpl" kindref="compound">VectorImpl</ref></computeroutput> is as easy as possible.</para>
     259             : <para>When a SpECTRE vector is constructed as owning, or becomes owning, it allocates its own block of memory of appropriate size, and stores a pointer to that memory in a <computeroutput><ref refid="cpp/memory/unique_ptr" kindref="compound" external="/__w/spectre/spectre/docs/config/cppreference-doxygen-web.tag.xml">std::unique_ptr</ref></computeroutput> named <computeroutput>owned_data_</computeroutput>. The <computeroutput><ref refid="cpp/memory/unique_ptr" kindref="compound" external="/__w/spectre/spectre/docs/config/cppreference-doxygen-web.tag.xml">std::unique_ptr</ref></computeroutput> ensures that the SpECTRE vector needs to perform no further direct memory management, and that the memory will be appropriately managed whenever the <computeroutput><ref refid="cpp/memory/unique_ptr" kindref="compound" external="/__w/spectre/spectre/docs/config/cppreference-doxygen-web.tag.xml">std::unique_ptr</ref> owned_data_</computeroutput> member is deleted or moved. The base <computeroutput><ref refid="structPointerVector" kindref="compound">PointerVector</ref></computeroutput> must also be told about the pointer, which is always accomplished by calling the protected function <computeroutput>VectorImpl.reset_pointer_vector(const size_t set_size)</computeroutput>, which sets the <computeroutput><ref refid="structPointerVector" kindref="compound">PointerVector</ref></computeroutput> internal pointer to the pointer obtained by <computeroutput>std::unique_pointer.get()</computeroutput>.</para>
     260             : <para>When a SpECTRE vector is constructed as non-owning by the <computeroutput><ref refid="classVectorImpl" kindref="compound">VectorImpl</ref>(ValueType* start, size_t set_size)</computeroutput> constructor, or becomes non-owning by the <computeroutput>set_data_ref</computeroutput> function, the internal <computeroutput><ref refid="cpp/memory/unique_ptr" kindref="compound" external="/__w/spectre/spectre/docs/config/cppreference-doxygen-web.tag.xml">std::unique_ptr</ref></computeroutput> named <computeroutput>owned_data_</computeroutput> no longer points to the data represented by the vector and can be thought of as &quot;inactive&quot; for the purposes of computation and memory management. This behavior is desirable, because otherwise the <computeroutput><ref refid="cpp/memory/unique_ptr" kindref="compound" external="/__w/spectre/spectre/docs/config/cppreference-doxygen-web.tag.xml">std::unique_ptr</ref></computeroutput> would attempt to free memory that is presumed to be also used elsewhere, causing difficult to diagnose memory errors. The non-owning SpECTRE vector updates the base <computeroutput><ref refid="structPointerVector" kindref="compound">PointerVector</ref></computeroutput> pointer directly by calling <computeroutput>PointerVector.reset</computeroutput> from the derived class (on itself). </para>
