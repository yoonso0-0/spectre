      45             : <sect1 id="travis_guide_1travis_ci">
      46             : <title>Testing SpECTRE with Travis CI</title>
      47             : <para>SpECTRE uses <ulink url="https://travis-ci.org/sxs-collaboration/spectre">TravisCI</ulink> for testing the code. Multiple build jobs (described below) are launched each time a pull request is submitted or updated. In addition, a daily cron job is run. Travis will also launch these build jobs each time you push to a branch on your fork of SpECTRE if you enable it. (Go to Travis, click on your name in the upper right corner, and then towards the bottom, enable testing on your fork of SpECTRE.)</para>
      48             : <para>For pull requests, you can view the Travis CI build by clicking on <computeroutput>Details</computeroutput> next to the <computeroutput>Travis build failed</computeroutput> or <computeroutput>Travis build succeeded</computeroutput> in the section for <computeroutput>All checks have failed</computeroutput> or <computeroutput>All checks have passed</computeroutput> towards the bottom of the pull request. You can login to Travis using your GitHub credentials in order to cancel your builds, and you should do so if you update your pull request while it is building.</para>

     106             : <sect2 id="travis_guide_1perform-checks-locally">
     107             : <title>How to perform the checks locally</title>
     108             : <para>Before pushing to GitHub and waiting for Travis to perform the checks it is useful to perform at least the following tests locally:<itemizedlist>
     109             : <listitem><para><bold>Unit tests:</bold> Perform a <computeroutput>make test-executables</computeroutput> and then execute <computeroutput>ctest</computeroutput> to run all unit tests. As for <computeroutput>make</computeroutput> you can append a <computeroutput>-jN</computeroutput> flag to <computeroutput>ctest</computeroutput> to run in parallel on <computeroutput>N</computeroutput> cores. To run only a subset of the tests you can use one of the other keywords that the tests are labeled with, such as <computeroutput>ctest -L datastructures</computeroutput>. To run only particular tests you can also execute <computeroutput>ctest -R TEST_NAME</computeroutput> instead, where <computeroutput>TEST_NAME</computeroutput> is a regular expression matching the test identifiers such as <computeroutput>Unit.DataStructures.Mesh</computeroutput>. Pass the flag <computeroutput>--output-on-failure</computeroutput> to get output from failed tests. Consult <computeroutput>ctest -h</computeroutput> for further options.</para>
     110             : </listitem><listitem><para><bold>clang-tidy:</bold> In a clang build directory, run <computeroutput>make clang-tidy FILE=SOURCE_FILE</computeroutput> where <computeroutput>SOURCE_FILE</computeroutput> is a relative or absolute path to a <computeroutput>.cpp</computeroutput> file. To perform this check for all source files that changed in your pull request, <computeroutput>make clang-tidy-hash HASH=UPSTREAM_HEAD</computeroutput> where <computeroutput>UPSTREAM_HEAD</computeroutput> is the hash of the commit that your pull request is based on, usually the <computeroutput>HEAD</computeroutput> of the <computeroutput>upstream/develop</computeroutput> branch.</para>
     111             : </listitem><listitem><para><bold>Documentation:</bold> To render the documentation for the current state of the source tree the command <computeroutput>make doc</computeroutput> (or <computeroutput>make doc-check</computeroutput> to highlight warnings) can be used, placing its result in the <computeroutput>docs</computeroutput> directory in the build tree. Once code has been made into a pull request to GitHub, the documentation can be rendered locally using the <computeroutput>tools/pr-docs</computeroutput> script. To view the documentation, simply open the <computeroutput>index.html</computeroutput> file in the <computeroutput>html</computeroutput> subdirectory in a browser. Some functionality requires a web server (e.g. citation popovers), so just run a <computeroutput>python3 -m http.server</computeroutput> in the <computeroutput>html</computeroutput> directory to enable this.</para>
     112             : </listitem><listitem><para><bold>IWYU:</bold> We experimented for a time using IWYU (include what you use) as an automated check for correct includes and forward declarations. Unfortunately it gave many incorrect suggestions. We have decided to no longer have a IWYU check for Travis, but have left support for IWYU so that it can be used locally. To do so just for the changed files in a pull request run <computeroutput>make iwyu-hash HASH=UPSTREAM_HEAD</computeroutput>. Since IWYU requires <computeroutput>USE_PCH=OFF</computeroutput> you can create a separate build directory and append <computeroutput>-D USE_PCH=OFF</computeroutput> to the usual <computeroutput>cmake</computeroutput> call. Note that it is very easy to incorrectly install IWYU (if not using the Docker container) and generate nonsense errors. Note that we have left IWYU pragmas in the code, but no longer require they be added so that IWYU gives no errors when run. As IWYU is still under development, we plan to investigate using it again in the future.</para>

     131             : <sect2 id="travis_guide_1precompiled-headers-ccache">
     132             : <title>Precompiled Headers and ccache</title>
     133             : <para>Getting ccache to work with precompiled headers on TravisCI is a little challenging. The header to be precompiled is <computeroutput>${SPECTRE_SOURCE_DIR}/tools/SpectrePch.hpp</computeroutput> and is symbolically linked to <computeroutput>${SPECTRE_BUILD_DIR}/SpectrePch.hpp</computeroutput>. The configuration that seems to work is specifying the environment variables:</para>
     134             : <para><programlisting filename=".sh"><codeline><highlight class="normal">CCACHE_COMPILERCHECK=content</highlight></codeline>
     135             : <codeline><highlight class="normal">CCACHE_EXTRAFILES=&quot;${SPECTRE_SOURCE_DIR}/tools/SpectrePch.hpp&quot;</highlight></codeline>
     136             : <codeline><highlight class="normal">CCACHE_IGNOREHEADERS=&quot;${SPECTRE_BUILD_DIR}/SpectrePch.hpp:${SPECTRE_BUILD_DIR}/SpectrePch.hpp.gch&quot;</highlight></codeline>
     137             : </programlisting></para>
     138             : <para>On macOS builds we haven&apos;t yet had success with using <computeroutput>ccache</computeroutput> with a precompiled header. We disable the precompiled header and build in debug mode only to have reasonable build times.</para>
     139             : </sect2>
     140             : <sect2 id="travis_guide_1travis-build-stages">
     141             : <title>Build Stages</title>
     142             : <para>In order to avoid timeouts we build SpECTRE in various stages, carrying over the ccache from one stage to the next. This allows us to avoid recompiling the code in the next stage (there is some small overhead from running ccache instead of not doing anything at all). The first stage builds all the SpECTRE libraries but none of the executables or testing libraries. The second stage builds builds the test executables, runs the tests, and also runs ClangTidy, include-what-you-use, and various other checks. Another stage could be added that builds some of the test libraries if necessary.</para>
     143             : </sect2>
     144             : <sect2 id="travis_guide_1caching-mac-os">
     145             : <title>Caching Dependencies on macOS Builds</title>
     146             : <para>On macOS builds we cache all of our dependencies, like LIBXSMM and Charm++. These are cached in <computeroutput>$HOME/mac_cache</computeroutput>. Ultimately this saves about 10-12 minutes even when compared to using ccache to cache the object files from building the dependencies. We also cache <computeroutput>$HOME/Library/Caches/Homebrew</computeroutput>, which is where Homebrew keeps the downloaded formulas. By caching the Homebrew bottles we are able to avoid brew formulas building from source because a tarball of the package was not available at the time. </para>
