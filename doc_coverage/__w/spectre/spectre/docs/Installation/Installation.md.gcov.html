      69             : ## Using Docker to obtain a SpECTRE environment
      70             : 
      71             : A [Docker](https://www.docker.com/) image is available from
      72             : [DockerHub](https://hub.docker.com/r/sxscollaboration/spectrebuildenv/) and can
      73             : be used to build SpECTRE on a personal machine.
      74             : 
      75             : **Note**: The Docker image is the recommended way of using SpECTRE on a personal
      76             : Linux machine. Because of the wide variety of operating systems available today
      77             : it is not possible for us to support all configurations. However, using Spack
      78             : as outlined below is a supported alternative to Docker images.
      79             : 
      80             : **NOTE**: If you have SELinux active
      81             : on your system you must figure out how to enable sharing files with the host
      82             : OS. If you receive errors that you do not have permission to access a shared
      83             : directory it is likely that your system has SELinux enabled. One option is to
      84             : disable SELinux at the expense of reducing the security of your system.
      85             : 
      86             : To build with the docker image:
      87             : 
      88             : 1. Clone SpECTRE into SPECTRE_ROOT, a directory of your choice.
      89             : 2. Retrieve the docker image (you may need `sudo` in front of this command)
      90             :    ```
      91             :    docker pull sxscollaboration/spectrebuildenv:latest
      92             :    ```
      93             : 3. Start the docker container (you may need `sudo`)
      94             :    ```
      95             :    docker run -v SPECTRE_ROOT:SPECTRE_ROOT --name CONTAINER_NAME \
      96             :               -i -t sxscollaboration/spectrebuildenv:latest /bin/bash
      97             :    ```
      98             :    - The `--name CONTAINER_NAME` is optional, where CONTAINER_NAME is a name
      99             :    of your choice. If you don't name your container, docker will generate an
     100             :    arbitrary name.
     101             :    - On macOS you can significantly increase the performance of file system
     102             :    operations by appending the flag `:delegated` to `-v`, e.g.
     103             :    `-v SPECTRE_ROOT:SPECTRE_ROOT:delegated` (see
     104             :    https://docs.docker.com/docker-for-mac/osxfs-caching/).
     105             :    - It can be useful to expose a port to the host so you can run servers such
     106             :    as [Jupyter](https://jupyter.org/index.html) for accessing the Python
     107             :    bindings (see \ref spectre_using_python) or a Python web server to view the
     108             :    documentation. To do so, append the `-p` option, e.g. `-p 8000:8000`.
     109             : 
     110             :    You will end up in a shell in the docker container,
     111             :    as root (you need to be root).
     112             :    Within the container, the files in SPECTRE_ROOT are available and
     113             :    Charm++ is installed in `/work/charm`. For the following steps, stay inside
     114             :    the docker container as root.
     115             : 4. Make a build directory somewhere inside the container, e.g.
     116             :    `/work/spectre-build-gcc`, and cd into it.
     117             : 5. Build SpECTRE with
     118             : ```
     119             :    cmake -D CMAKE_Fortran_COMPILER=gfortran-8 \
     120             :          -D CHARM_ROOT=/work/charm/multicore-linux-x86_64-gcc SPECTRE_ROOT
     121             : ```
     122             :    then
     123             :    `make -jN`
     124             :    to compile the code, `make test-executables -jN` to compile the test
     125             :    executables, and `ctest` to run the tests.
     126             : 
     127             : Notes:
     128             :   * Everything in your build directory is owned by root, and is
     129             :     accessible only within the container.
     130             :   * You should edit source files in SPECTRE_ROOT in a separate terminal
     131             :     outside the container, and use the container only for compiling and
     132             :     running the code.
     133             :   * If you exit the container (e.g. ctrl-d),
     134             :     your compilation directories are still saved, as is the patch
     135             :     that you have applied to /work/charm and any other changes to
     136             :     the container that you have made.
     137             :     To restart the container, try the following commands
     138             :     (you may need `sudo`):
     139             :     1. `docker ps -a`,
     140             :       to list all containers with their CONTAINER_IDs and CONTAINER_NAMEs,
     141             :     2. `docker start -i CONTAINER_NAME` or `docker start -i CONTAINER_ID`,
     142             :       to restart your container.
     143             :   * You can run more than one shell in the same container, for instance
     144             :     one shell for compiling with gcc and another for compiling
     145             :     with clang.
     146             :     To add a new shell, run `docker exec -it CONTAINER_NAME /bin/bash`
     147             :     (or `docker exec -it CONTAINER_ID /bin/bash`) from
     148             :     a terminal outside the container.
     149             :   * In step 3 above, the `-v SPECTRE_ROOT:SPECTRE_ROOT` maps the directory
     150             :     SPECTRE_ROOT outside the container to SPECTRE_ROOT inside the container.
     151             :     Technically docker allows you to say `-v SPECTRE_ROOT:/my/new/path`
     152             :     to map SPECTRE_ROOT outside the container to any path you want inside
     153             :     the container, but **do not do this**.  Compiling inside the container
     154             :     sets up git hooks in SPECTRE_ROOT that
     155             :     contain hardcoded pathnames to SPECTRE_ROOT *as seen from
     156             :     inside the container*. So if your source paths inside and outside the
     157             :     container are different, commands like `git commit` run *from
     158             :     outside the container* will die with `No such file or directory`.
     159             :   * To compile with clang, the cmake command is
     160             : ```
     161             :     cmake -D CMAKE_CXX_COMPILER=clang++ \
     162             :           -D CMAKE_C_COMPILER=clang \
     163             :           -D CMAKE_Fortran_COMPILER=gfortran-8 \
     164             :           -D CHARM_ROOT=/work/charm/multicore-linux-x86_64-clang SPECTRE_ROOT
     165             : ```
     166             :   * To compile the Python bindings, add the option
     167             :     `-D BUILD_PYTHON_BINDINGS=ON` to the `cmake` command (see
     168             :     \ref spectre_writing_python_bindings). You can specify the Python version,
     169             :     interpreter and libraries used for compiling and testing the bindings by
     170             :     setting the `-D Python_EXECUTABLE` to an absolute path such as
     171             :     `/usr/bin/python3`.
     172             :   * When the Docker container gets updated, you can stop it with
     173             :     `docker stop CONTAINER_NAME`, remove it with `docker rm CONTAINER_NAME`
     174             :     and then start at step 2 above to run it again.

     176             : ## Using Singularity to obtain a SpECTRE environment
     177             : 
     178             : [Singularity](https://sylabs.io) is a container alternative
     179             : to Docker with better security and nicer integration.
     180             : 
     181             : To use Singularity you must:
     182             : 
     183             : 1. Build [Singularity](https://sylabs.io) and add it to your
     184             :    `$PATH`
     185             : 2. `cd` to the directory where you want to store the SpECTRE Singularity image,
     186             :    source, and build directories, let's call it WORKDIR. The WORKDIR must be
     187             :    somewhere in your home directory. If this does not work for you, follow the
     188             :    Singularity instructions on setting up additional [bind
     189             :    points](https://sylabs.io/guides/3.7/user-guide/bind_paths_and_mounts.html) 
     190             :    (version 3.7. For other versions, see the [docs](https://sylabs.io/docs/)). 
     191             :    Once inside the WORKDIR, clone SpECTRE into `WORKDIR/SPECTRE_ROOT`.
     192             : 3. Run `sudo singularity build spectre.img
     193             :    docker://sxscollaboration/spectrebuildenv:latest`.
     194             : 
     195             :    If you get the error message that `makesquashfs` did not have enough space to
     196             :    create the image you need to set a different `SINGULARITY_TMPDIR`. This can
     197             :    be done by running: `sudo SINGULARITY_TMPDIR=/path/to/new/tmp singularity
     198             :    build spectre.img docker://sxscollaboration/spectrebuildenv:latest`. Normally
     199             :    `SINGULARITY_TMPDIR` is `/tmp`, but building the image will temporarily need
     200             :    almost 8GB of space.
     201             : 
     202             :    You can control where Singularity stores the downloaded image files from
     203             :    DockerHub by specifying the `SINGULARITY_CACHEDIR` environment variable. The
     204             :    default is `$HOME/.singularity/`. Note that `$HOME` is `/root` when running
     205             :    using `sudo`.
     206             : 4. To start the container run `singularity shell spectre.img` and you
     207             :    will be dropped into a bash shell.
     208             : 5. Run `cd SPECTRE_HOME && mkdir build && cd build` to set up a build
     209             :    directory.
     210             : 6. To build SpECTRE run
     211             : ```
     212             :    cmake -D CMAKE_Fortran_COMPILER=gfortran-8 \
     213             :          -D CHARM_ROOT=/work/charm/multicore-linux-x86_64-gcc SPECTRE_ROOT
     214             : ```
     215             :    followed by
     216             :    `make -jN` where `N` is the number of cores to build on in parallel.
     217             : 
     218             : Notes:
     219             : - You should edit source files in SPECTRE_ROOT in a separate terminal outside
     220             :   the container, and use the container only for compiling and running the code.
     221             : - Unlike Docker, Singularity does not keep the state between runs. However, it
     222             :   shares the home directory with the host OS so you should do all your work
     223             :   somewhere in your home directory.
     224             : - To run more than one container just do `singularity shell spectre.img` in
     225             :   another terminal.
     226             : - Since the data you modify lives on the host OS there is no need to worry about
     227             :   losing any data, needing to clean up old containers, or sharing data between
     228             :   containers and the host.
     229             : - To build with clang, the CMake command is:
     230             : ```
     231             :     cmake -D CMAKE_CXX_COMPILER=clang++ \
     232             :           -D CMAKE_C_COMPILER=clang \
     233             :           -D CMAKE_Fortran_COMPILER=gfortran-8 \
     234             :           -D CHARM_ROOT=/work/charm/multicore-linux-x86_64-clang SPECTRE_ROOT
     235             : ```

     237             : ## Using Spack to set up a SpECTRE environment
     238             : 
     239             : SpECTRE's dependencies can be installed with
     240             : [Spack](https://github.com/LLNL/spack), a package manager tailored for HPC use.
     241             : Install Spack by cloning it into `SPACK_DIR` (a directory of your choice),
     242             : then add `SPACK_DIR/bin` to your `PATH`.
     243             : 
     244             : For security, it is good practice to make Spack use the system's OpenSSL
     245             : rather than allow it to install a new copy — see Spack's documentation for
     246             : [instructions](https://spack.readthedocs.io/en/latest/getting_started.html#openssl).
     247             : You may need to install the development version of OpenSSL:
     248             : * On Ubuntu (16.04), run `sudo apt-get install libssl-dev`
     249             : * On Fedora (27), run `sudo dnf install openssl-devel`
     250             : 
     251             : Spack works well with a module environment. We recommend
     252             : [LMod](https://github.com/TACC/Lmod), which is available on many systems:
     253             : * On macOS, install LMod from [brew](https://brew.sh/), then source the LMod
     254             :   shell script by adding `. /usr/local/Cellar/lmod/YOUR_VERSION_NUMBER/init/sh`
     255             :   to your `.bash_profile`.
     256             : * On Ubuntu, run `sudo apt-get install -y lmod` and, for Ubuntu 17.04, add
     257             :   `. /etc/profile.d/lmod.sh` to your `.bashrc`. For Ubuntu 16.04, the correct
     258             :   path to add is `. /usr/share/lmod/lmod/init/bash`.
     259             : * On Arch Linux, run `yaourt -Sy lmod` and add `. /etc/profile.d/lmod.sh` to
     260             :   your `.bashrc`,
     261             : * On Fedora/RHEL, GNU Environment Modules comes out-of-the-box and works equally
     262             :   well.
     263             : * Instructions for other Linux distros are available online.
     264             : 
     265             : To use modules with Spack, enable Spack's shell support by adding
     266             : `. SPACK_DIR/share/spack/setup-env.sh` to your `.bash_profile` or `.bashrc`.
     267             : 
     268             : Once you have Spack installed and configured with OpenSSL and LMod, you can
     269             : install the SpECTRE dependencies using
     270             : ```
     271             : spack install blaze@3.2
     272             : spack install brigand@master
     273             : spack install libsharp -openmp -mpi
     274             : spack install catch2
     275             : spack install gsl
     276             : spack install jemalloc # or from your package manager
     277             : spack install libxsmm
     278             : spack install yaml-cpp@develop
     279             : ```
     280             : You can also install CMake, OpenBLAS, Boost, and HDF5 from Spack.
     281             : To load the packages you've installed from Spack run `spack load PACKAGE`,
     282             : or (equivalently) use the `module load` command.
     283             : 
     284             : **Note**: Spack allows very flexible configurations and
     285             : it is recommended you read the [documentation](https://spack.readthedocs.io) if
     286             : you require features such as packages installed with different compilers.
     287             : 
     288             : **Note**: On a Mac, you may need to `spack install
     289             : yaml-cpp@develop~shared` (note that is a tilde and not a dash in front
     290             : of shared) in order to force the building of the static libraries in
     291             : order to avoid dynamic linking errors.
     292             : 
     293             : ### Building SpECTRE
     294             : 
     295             : After the dependencies have been installed, Charm++ and SpECTRE can be compiled.
     296             : Follow these steps:
     297             : 
     298             : 1.  Clone [SpECTRE](https://github.com/sxs-collaboration/spectre) into
     299             :     `SPECTRE_ROOT`, a directory of your choice.
     300             : 2.  Install Charm++:
     301             :   * Clone [Charm++](http://charm.cs.illinois.edu/software) into `CHARM_DIR`,
     302             :     again a directory of your choice.
     303             :   * In `CHARM_DIR`, run
     304             :     `git checkout v6.10.2` to switch to a supported, stable release of Charm++.
     305             :   * Charm++ is compiled by running
     306             :     `./build charm++ ARCH OPTIONS`.
     307             :     To figure out the correct target architecture and options, you can simply
     308             :     run `./build`; the script will then ask you questions to guide you towards
     309             :     the correct settings (see notes below for additional details).
     310             :     Then compile Charm++.
     311             :     The Charm++ build will be located in a new directory,
     312             :     `CHARM_DIR/ARCH_OPTS`, whose name may (or may not) have some of the options
     313             :     appended to the architecture.
     314             :   * On macOS 10.12 it is necessary to patch the STL implementation. Insert
     315             :     \code
     316             :     #ifndef _MACH_PORT_T
     317             :     #define _MACH_PORT_T
     318             :     #include <sys/_types.h> /* __darwin_mach_port_t */
     319             :     typedef __darwin_mach_port_t mach_port_t;
     320             :     #include <pthread.h>
     321             :     mach_port_t pthread_mach_thread_np(pthread_t);
     322             :     #endif /* _MACH_PORT_T */
     323             :     \endcode
     324             :     into the front of
     325             :     `/Library/Developer/CommandLineTools/usr/include/c++/v1/__threading_support`
     326             : 3.  Return to `SPECTRE_ROOT`, and create a build dir by running
     327             :     `mkdir build && cd build`
     328             : 4.  Build SpECTRE with
     329             :     `cmake -D CHARM_ROOT=CHARM_DIR/ARCH_OPTS SPECTRE_ROOT`
     330             :     then
     331             :     `make -jN`
     332             :     to compile the code.
     333             : 5.  Run the tests with
     334             :     `make test-executables && ctest`.
     335             : 
     336             : **Notes**:
     337             : * For more details on building Charm++, see the directions
     338             :   [here](http://charm.cs.illinois.edu/manuals/html/charm++/A.html)
     339             :   The correct target is `charm++` and, for a personal machine, the
     340             :   correct target architecture is likely to be `multicore-linux-x86_64`
     341             :   (or `multicore-darwin-x86_64` on macOS).
     342             :   On an HPC system, the correct Charm++ target architecture depends on the
     343             :   machine's inter-node communication architecture. We will be providing specific
     344             :   instructions for various HPC systems.
     345             : * Both Charm++ and SpECTRE must be compiled using the same compiler,
     346             :   otherwise you will receive undefined reference errors while linking SpECTRE.
     347             :   When compiling Charm++ you can specify the compiler using, for example,
     348             :   ```
     349             :   ./build charm++ ARCH clang
     350             :   ```
     351             :   When compiling SpECTRE you can specify the compiler to CMake using,
     352             :   for example,
     353             :   ```
     354             :   cmake -D CMAKE_CXX_COMPILER=clang++ \
     355             :         -D CMAKE_C_COMPILER=clang \
     356             :         -D CMAKE_Fortran_COMPILER=gfortran \
     357             :         -D CHARM_ROOT=CHARM_DIR/ARCH_OPTS SPECTRE_ROOT
     358             :   ```
     359             : * Inside the SpECTRE build directory, use `make list` to see all available
     360             :   targets. This list can be refreshed by running CMake again.
     361             :
