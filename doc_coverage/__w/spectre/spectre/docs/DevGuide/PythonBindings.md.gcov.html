       5             : # Writing Python Bindings {#spectre_writing_python_bindings}
       6             : 
       7             : ## CMake and Directory Layout
       8             : 
       9             : To allow users to analyze output from simulations and take advantage of
      10             : SpECTRE's data structures and functions in python, bindings must sometimes be
      11             : written. SpECTRE uses [pybind11](https://pybind11.readthedocs.io/)
      12             : to aid with generating the bindings. The C++ code for the bindings should
      13             : generally go in a `Python` subdirectory. For example, the bindings for the
      14             : DataStructures library would go in `src/DataStructures/Python/`. SpECTRE
      15             : provides the `spectre_python_add_module` CMake function to make adding a new
      16             : python module, be it with or without bindings, easy.  The python bindings are
      17             : built only if `-D BUILD_PYTHON_BINDINGS=ON` is passed when invoking cmake.
      18             : You can specify the Python version, interpreter and libraries used for compiling
      19             : and testing the bindings by setting the `-D Python_EXECUTABLE` to an absolute
      20             : path such as `/usr/bin/python3`.
      21             : 
      22             : The function `spectre_python_add_module` takes as its first argument the module,
      23             : in our case `DataStructures`. Optionally, a list of `SOURCES` can be passed to
      24             : the CMake function. If you specify `SOURCES`, you must also specify a
      25             : `LIBRARY_NAME`. A good `LIBRARY_NAME` is the name of the C++ library for which
      26             : bindings are being built prefixed with `Py`, e.g. `PyDataStructures`. If the
      27             : Python module will only consist of Python files, then the `SOURCES` option
      28             : should not be specified. Python files that should be part of the module can be
      29             : passed with the keywords `PYTHON_FILES` or `PYTHON_EXECUTABLES`, where the
      30             : latter exposes them in `bin/` in the SpECTRE build directory. For example, the
      31             : arguments `PYTHON_FILES Hello.py PYTHON_EXECUTABLES HelloWorld.py` produce a
      32             : Python module that contains both files, but `HelloWorld.py` can be run directly
      33             : as `bin/HelloWorld` from the SpECTRE build directory. Finally, the `MODULE_PATH`
      34             : named argument can be passed with a string that is the path to where the module
      35             : should be. For example, `MODULE_PATH "submodule0/submodule1/"` would mean the
      36             : module is accessed from python using
      37             : `import spectre.submodule0.submodule1.MODULE_NAME`.
      38             : 
      39             : Here is a complete example of how to call the `spectre_python_add_module`
      40             : function:
      41             : 
      42             : \code
      43             : spectre_python_add_module(
      44             :   Extra
      45             :   LIBRARY_NAME "PyExtraDataStructures"
      46             :   MODULE_PATH "DataStructures/"
      47             :   SOURCES Bindings.cpp MyCoolDataStructure.cpp
      48             :   PYTHON_FILES CoolPythonDataStructure.py
      49             :   PYTHON_EXECUTABLES HelloWorld.py
      50             :   )
      51             : \endcode
      52             : 
      53             : The library that is added has the name `PyExtraDataStructures`. Make sure to
      54             : call `spectre_python_link_libraries` for every Python module that compiles
      55             : `SOURCES`. For example,
      56             : 
      57             : \code
      58             : spectre_python_link_libraries(
      59             :   PyExtraDataStructures
      60             :   PRIVATE
      61             :   ExtraDataStructures
      62             :   pybind11::module
      63             :   )
      64             : \endcode
      65             : 
      66             : You may also call `spectre_python_add_dependencies` for Python modules that
      67             : have `SOURCES`, e.g.
      68             : 
      69             : \code
      70             : spectre_python_add_dependencies(
      71             :   PyExtraDataStructures
      72             :   PyDataStructures
      73             :   )
      74             : \endcode
      75             : 
      76             : Note that these functions will skip adding or configure any C++ libraries if
      77             : the `BUILD_PYTHON_BINDINGS` flag is `OFF`.
      78             : 
      79             : ## Writing Bindings
      80             : 
      81             : Once a python module has been added you can write the actual bindings. You
      82             : should structure your bindings directory to reflect the structure of the library
      83             : you're writing bindings for. For example, say we want bindings for `DataVector`
      84             : and `Matrix` then we should have one source file for each class's bindings
      85             : inside `src/DataStructures/Python`. The functions that generate the bindings
      86             : should be in the `py_bindings` namespace and have a reasonable name such as
      87             : `bind_datavector`. There should be a file named `Bindings.cpp` which calls all
      88             : the `bind_*` functions. The `Bindings.cpp` file is quite simple and should
      89             : `include <pybind11/pybind11.h>`, forward declare the `bind_*` functions, and
      90             : then have a `PYBIND11_MODULE` function. For example,
      91             : 
      92             : \code{.cpp}
      93             : #include <pybind11/pybind11.h>
      94             : 
      95             : namespace py = pybind11;
      96             : 
      97             : namespace py_bindings {
      98             : void bind_datavector(py::module& m);
      99             : }  // namespace py_bindings
     100             : 
     101             : PYBIND11_MODULE(_PyDataStructures, m) {
     102             :   py_bindings::bind_datavector(m);
     103             : }
     104             : \endcode
     105             : 
     106             : Note that the library name is passed to `PYBIND11_MODULE` and is prefixed
     107             : with an underscore. The underscore is important and the library name must be the
     108             : same that is passed as `LIBRARY_NAME` to `spectre_python_add_module` (see
     109             : above).
     110             : 
     111             : The `DataVector` bindings serve as an example with code comments on how to write
     112             : bindings for a class. There is also extensive documentation available directly
     113             : from [pybind11](https://pybind11.readthedocs.io/). SpECTRE currently aims to
     114             : support both Python 2.7 and Python 3 and as such all bindings must support both.
     115             : 
     116             : \note Exceptions should be allowed to propagate through the bindings so that
     117             : error handling via exceptions is possible from python rather than having the
     118             : python interpreter being killed with a call to `abort`.
     119             : 
     120             : ## Testing Python Bindings and Code
     121             : 
     122             : All the python bindings must be tested. SpECTRE uses the
     123             : [unittest](https://docs.python.org/3/library/unittest.html) framework
     124             : provided as part of python. To register a test file with CMake use the
     125             : SpECTRE-provided function `spectre_add_python_test` passing as the first
     126             : argument the test name (e.g. `"Unit.DataStructures.Python.DataVector"`), the
     127             : file as the second argument (e.g. `Test_DataVector.py`), and a semicolon
     128             : separated list of labels as the last (e.g. `"unit;datastructures;python"`).
     129             : All the test cases should be in a single class so that the python unit testing
     130             : framework will run all test functions on a single invocation to avoid startup
     131             : cost.
     132             : 
     133             : Below is an example of registering a python test file for bindings:
     134             : 
     135             : \snippet tests/Unit/DataStructures/CMakeLists.txt example_add_pybindings_test
     136             : 
     137             : Python code that does not use bindings must also be tested. You can register the
     138             : test file using the `spectre_add_python_test` CMake function with the same
     139             : signature as shown above.
     140             : 
     141             : Please note that the tests must be formatted according to the `.style.yapf` file
     142             : in the root of the repository.
     143             : 
     144             : ## Using The Bindings
     145             : 
     146             : See \ref spectre_using_python "Using SpECTRE's Python"
     147             : 
     148             : ## Notes:
     149             : 
     150             : - All python libraries are dynamic/shared libraries.
     151             : - Exceptions should be allowed to propagate through the bindings so that
     152             :   error handling via exceptions is possible from python rather than having the
     153             :   python interpreter being killed with a call to `abort`.
     154             : - All function arguments in Python bindings should be named using `py::arg`.
     155             :   See the Python bindings in `IO/H5/` for examples. Using the named arguments in
     156             :   Python code is optional, but preferred when it makes code more readable.
     157             :   In particular, use the argument names in the tests for the Python bindings so
     158             :   they are being tested as well.
