       5             : # SFINAE {#sfinae}
       6             : 
       7             : SFINAE, Substitution Failure Is Not An Error, means that if a deduced template
       8             : substitution fails, compilation must continue. This can be exploited to make
       9             : decisions at compile time. See [here](http://nilsdeppe.com/posts/tmpl-part1)
      10             : for a discussion using `std::enable_if` to remove certain functions from
      11             : overload resolution or certain template specializations from name lookup.
      12             : Another method of controlling name lookup resolution is using
      13             : `std::void_t`. `void_t` is a metafunction from types to `void`, that is
      14             : 
      15             : ```cpp
      16             : template <typename... Args>
      17             : using void_t = void;
      18             : ```
      19             : 
      20             : `void_t` is useful when used in combination with `decltype` and `std::declval`
      21             : to probe if a type has certain members. For example, we can implement a type
      22             : trait to check if a type `T` is iterable by first have the general definition
      23             : inherit from `std::false_type` as follows,
      24             : 
      25             : ```cpp
      26             : template <typename T, typename = void>
      27             : struct is_iterable : std::false_type {};
      28             : ```
      29             : 
      30             : Next we will have specialization that uses `void_t` to check if the type `T`
      31             : has a `begin()` and `end()` function.
      32             : 
      33             : ```cpp
      34             : template <typename T>
      35             : struct is_iterable<T, std::void_t<decltype(std::declval<T>().begin(),
      36             :                                            std::declval<T>().end())>>
      37             :     : std::true_type {};
      38             : ```
      39             : 
      40             : What is happening here? Well, we use `std::declval` to convert the type `T`
      41             : to a reference type, which allows us to call member functions inside `decltype`
      42             : expressions without having to construct an object. First we try to call the
      43             : member function `begin()` on `std::declval<T>()`, and if that succeeds we
      44             : throw away the result using the comma operator. Next we try to call `end()`
      45             : on `std::declval<T>()`, which, if it succeeds we get the return type of
      46             : using `decltype`. Note that `decltype` is important because we can only call
      47             : member functions on reference types inside of `decltype`, not just anywhere.
      48             : Finally, if all this succeeded use `void_t` to metareturn `void`, otherwise
      49             : the template parameters of `void_t` fail to evaluate and the specialization
      50             : cannot be resolved during name lookup. We could just as well use
      51             : 
      52             : ```cpp
      53             : template <typename T>
      54             : struct is_iterable<T, std::void_t<decltype(std::declval<T>().begin()),
      55             :                                   decltype(std::declval<T>().end())>>
      56             :     : std::true_type {};
      57             : ```
      58             : 
      59             : Which of the two implementations of the `is_iterable` is preferred is simply
      60             : a matter of taste, both behave correctly.
      61             : 
      62             : If you're reading closely you might wonder why the `void_t` is necessary at
      63             : all, why not just `decltype(...)`? Well the reason is that since the default
      64             : template parameter metavalue is `void`, the specialization cannot be resolved
      65             : during name lookup unless the second template parameter in the specialization
      66             : is either `void` as well or is explicitly specified when the class template
      67             : is being invoked. Thus, the clearest implementation probably is
      68             : 
      69             : ```cpp
      70             : template <typename T, typename = std::void_t<>>
      71             : struct is_iterable : std::false_type {};
      72             : 
      73             : template <typename T>
      74             : struct is_iterable<T, std::void_t<decltype(std::declval<T>().begin(),
      75             :                                            std::declval<T>().end())>>
      76             :     : std::true_type {};
      77             : ```
      78             : 
      79             : You could now also define a helper type alias and constexpr boolean
      80             : 
      81             : ```cpp
      82             : template <typename T>
      83             : using is_iterable_t = typename is_iterable<T>::type;
      84             : 
      85             : template <typename T>
      86             : constexpr bool is_iterable_v = is_iterable<T>::value;
      87             : ```
