       5             : # Parallelization in SpECTRE {#tutorial_parallel_concepts}
       6             : 
       7             : This overview describes the concepts and terminology that SpECTRE uses
       8             : to enable parallelism.  This overview is a general discussion with no
       9             : code examples.  Subsequent tutorials will provide a more in-depth
      10             : exploration of the parallelization infrastructure, including code
      11             : examples.
      12             : 
      13             : Unlike many parallel scientific codes which use data-based
      14             : parallelism, SpECTRE uses task-based parallelism.  The classical
      15             : strategy for parallelism (data-based parallelism) is to assign a
      16             : portion of the data to processes (or threads) that synchronously
      17             : execute compute kernels.  This is implemented in many codes but it is
      18             : difficult to design codes with this strategy that will efficiently
      19             : scale for complex multi-scale, multi-physics workloads. Task-based
      20             : parallelism provides a solution: Instead of dividing work between
      21             : parallel processes based on data ownership, there is a set of tasks
      22             : and their inter-dependencies. Tasks are scheduled and assigned to
      23             : processes dynamically, providing opportunities for load balancing and
      24             : minimization of idle threads.  By dividing the program into small
      25             : enough tasks such that you have several tasks per thread,
      26             : communication time is hidden by interleaving tasks that are ready to
      27             : be executed with tasks that are waiting for data.
      28             : 
      29             : In order to implement task-based parallelism, SpECTRE is built on top
      30             : of the parallel programming framework of the Charm++ library, which is
      31             : developed by the [Parallel Programming
      32             : Laboratory](http://charm.cs.illinois.edu/) at the University of
      33             : Illinois.  Charm++ is a mature parallel programming framework that
      34             : provides intra-node threading and can use a variety of communication
      35             : interfaces (including MPI) to communicate between nodes.  Charm++ has
      36             : a large user base, which includes users of the cosmological
      37             : \f$N\f$-body code
      38             : [ChaNGa](https://github.com/N-BodyShop/changa/wiki/ChaNGa) and of the
      39             : molecular dynamics code
      40             : [NAMD](https://www.ks.uiuc.edu/Research/namd/).
      41             : 
      42             : ## Charm++ basic concepts
      43             : 
      44             : In order to understand how parallelization works in SpECTRE, it is
      45             : useful to understand the basic concepts in the design of Charm++.
      46             : Much of the following is quoted verbatim from the [Charm++
      47             : documentation](https://charm.readthedocs.io), interspersed with
      48             : comments on how SpECTRE interacts with Charm++.
      49             : 
      50             : > Charm++ is a C++-based parallel programming system, founded on the
      51             : > migratable-objects programming model, and supported by a novel and
      52             : > powerful adaptive runtime system. It supports both irregular as well
      53             : > as regular applications, and can be used to specify task-parallelism
      54             : > as well as data parallelism in a single application. It automates
      55             : > dynamic load balancing for task-parallel as well as data-parallel
      56             : > applications, via separate suites of load-balancing strategies. Via
      57             : > its message-driven execution model, it supports automatic latency
      58             : > tolerance, modularity and parallel composition. Charm++ also supports
      59             : > automatic checkpoint/restart, as well as fault tolerance based on
      60             : > distributed checkpoints.
      61             : 
      62             : SpECTRE currently wraps only some of the features of Charm++,
      63             : primarily the ones that support task-parallelism.  We are just
      64             : beginning our exploration of dynamic load balancing.  Coming soon we
      65             : will utilize automatic checkpoint/restart.  At present we do not use
      66             : Charm++ support for fault tolerance.
      67             : 
      68             : > The key feature of the migratable-objects programming model is
      69             : > over-decomposition: The programmer decomposes the program into a
      70             : > large number of work units and data units, and specifies the
      71             : > computation in terms of creation of and interactions between these
      72             : > units, without any direct reference to the processor on which any
      73             : > unit resides. This empowers the runtime system to assign units to
      74             : > processors, and to change the assignment at runtime as necessary.
      75             : 
      76             : SpECTRE's parallelization module is designed to make it easy for users
      77             : to exploit the migratable-object model by providing a framework to
      78             : define the units into which a program can be decomposed.
      79             : 
      80             : > A basic unit of parallel computation in Charm++ programs is a
      81             : > chare.  At its most basic level, it is just a C++ object. A Charm++
      82             : > computation consists of a large number of chares distributed on
      83             : > available processors of the system, and interacting with each other
      84             : > via asynchronous method invocations. Asynchronously invoking a
      85             : > method on a remote object can also be thought of as sending a
      86             : > “message” to it. So, these method invocations are sometimes referred
      87             : > to as messages. (besides, in the implementation, the method
      88             : > invocations are packaged as messages anyway). Chares can be created
      89             : > dynamically.
      90             : 
      91             : In SpECTRE, we wrap Charm++ chares in struct templates that we call
      92             : __parallel components__ that represent a collection of distributed
      93             : objects.  We wrap the asynchronous method invocations between the
      94             : elements of parallel components in struct templates that we call
      95             : __actions__.  Thus, each element of a parallel component can be
      96             : thought of as a C++ object that exists on one core on the
      97             : supercomputer, and an action as calling a member function of that
      98             : object, even if the caller is on another core.
      99             : 
     100             : > Conceptually, the system maintains a “work-pool” consisting of seeds
     101             : > for new chares, and messages for existing chares. The Charm++
     102             : > runtime system (Charm RTS) may pick multiple items,
     103             : > non-deterministically, from this pool and execute them, with the
     104             : > proviso that two different methods cannot be simultaneously
     105             : > executing on the same chare object (say, on different
     106             : > processors). Although one can define a reasonable theoretical
     107             : > operational semantics of Charm++ in this fashion, a more practical
     108             : > description of execution is useful to understand Charm++. A Charm++
     109             : > application’s execution is distributed among Processing Elements
     110             : > (PEs), which are OS threads or processes depending on the selected
     111             : > Charm++ build options. On each PE, there is a scheduler operating
     112             : > with its own private pool of messages. Each instantiated chare has
     113             : > one PE which is where it currently resides. The pool on each PE
     114             : > includes messages meant for chares residing on that PE, and seeds
     115             : > for new chares that are tentatively meant to be instantiated on that
     116             : > PE. The scheduler picks a message, creates a new chare if the
     117             : > message is a seed (i.e. a constructor invocation) for a new chare,
     118             : > and invokes the method specified by the message. When the method
     119             : > returns control back to the scheduler, it repeats the cycle
     120             : > (i.e. there is no pre-emptive scheduling of other invocations).
     121             : 
     122             : It is very important to keep in mind that the actions that are
     123             : executed on elements of parallel components are done so
     124             : non-deterministically by the run-time system.  Therefore it is the
     125             : responsibility of the programmer to ensure that actions are not called
     126             : out of order.  This means that if action B must be executed after
     127             : action A on a given element of a parallel component, the programmer
     128             : must ensure that either action B is called after the completion of
     129             : action A (i.e. it is not sufficient that action B is invoked after
     130             : action A is invoked), or that a `is_ready` function of action B only
     131             : succeeds if action A has been completed.
     132             : 
     133             : > When a chare method executes, it may create method invocations for
     134             : > other chares. The Charm Runtime System (RTS) locates the PE where
     135             : > the targeted chare resides, and delivers the invocation to the
     136             : > scheduler on that PE.
     137             : 
     138             : In SpECTRE, this is done by one element of a parallel component
     139             : calling an action on an element of a another parallel component.
     140             : 
     141             : > Methods of a chare that can be remotely invoked are called entry
     142             : > methods. Entry methods may take serializable parameters, or a
     143             : > pointer to a message object. Since chares can be created on remote
     144             : > processors, obviously some constructor of a chare needs to be an
     145             : > entry method. Ordinary entry methods are completely non-preemptive-
     146             : > Charm++ will not interrupt an executing method to start any other
     147             : > work, and all calls made are asynchronous.
     148             : 
     149             : In SpECTRE, the struct template that defines a parallel component has
     150             : taken care of creating the entry methods for the underlying chare,
     151             : which are then called by invoking actions.
     152             : 
     153             : > Charm++ provides dynamic seed-based load balancing. Thus location
     154             : > (processor number) need not be specified while creating a remote
     155             : > chare. The Charm RTS will then place the remote chare on a suitable
     156             : > processor. Thus one can imagine chare creation as generating only a
     157             : > seed for the new chare, which may take root on some specific
     158             : > processor at a later time.
     159             : 
     160             : We are just in the process of beginning to explore the load-balancing
     161             : features of Charm++, but plan to have new elements of parallel
     162             : components (the wrapped chares) be creatable using actions, without
     163             : specifying the location on which the new element is created.
     164             : 
     165             : > Chares can be grouped into collections. The types of collections of
     166             : > chares supported in Charm++ are: chare-arrays, chare-groups, and
     167             : > chare-nodegroups, referred to as arrays, groups, and nodegroups
     168             : > throughout this manual for brevity. A Chare-array is a collection of
     169             : > an arbitrary number of migratable chares, indexed by some index
     170             : > type, and mapped to processors according to a user-defined map
     171             : > group. A group (nodegroup) is a collection of chares, with exactly
     172             : > one member element on each PE (“node”).
     173             : 
     174             : Each of SpECTRE's parallel components has a type alias `chare_type`
     175             : corresponding to whether it is a chare-array, chare-group, or
     176             : chare-nodegroup.  In addition we support a singleton which is
     177             : essentially a one-element array.
     178             : 
     179             : > Charm++ does not allow global variables, except readonly
     180             : > variables. A chare can normally only access its own data
     181             : > directly. However, each chare is accessible by a globally valid
     182             : > name. So, one can think of Charm++ as supporting a global object
     183             : > space.
     184             : 
     185             : SpECTRE does not use the readonly global variables provided by
     186             : Charm++.  Instead SpECTRE provides a nodegroup called the
     187             : `GlobalCache` which provides global access to both read-only
     188             : and writable objects, as well as a way to access every parallel component.
     189             : 
     190             : > Every Charm++ program must have at least one mainchare. Each
     191             : > mainchare is created by the system on processor 0 when the Charm++
     192             : > program starts up. Execution of a Charm++ program begins with the
     193             : > Charm RTS constructing all the designated mainchares. For a
     194             : > mainchare named X, execution starts at constructor X() or X(CkArgMsg
     195             : > *) which are equivalent. Typically, the mainchare constructor starts
     196             : > the computation by creating arrays, other chares, and groups. It can
     197             : > also be used to initialize shared readonly objects.
     198             : 
     199             : SpECTRE provides a pre-defined mainchare called `Main` that is run
     200             : when a SpECTRE executable is started.  `Main` will create the other
     201             : parallel components, and initialize items in the `GlobalCache`
     202             : whose items can be used by any parallel component.
     203             : 
     204             : > Charm++ program execution is terminated by the CkExit call. Like the
     205             : > exit system call, CkExit never returns, and it optionally accepts an
     206             : > integer value to specify the exit code that is returned to the
     207             : > calling shell. If no exit code is specified, a value of zero
     208             : > (indicating successful execution) is returned. The Charm RTS ensures
     209             : > that no more messages are processed and no entry methods are called
     210             : > after a CkExit. CkExit need not be called on all processors; it is
     211             : > enough to call it from just one processor at the end of the
     212             : > computation.
     213             : 
     214             : SpECTRE wraps `CkExit` with the function `Parallel::exit`.  As no more
     215             : messages are processed by Charm++ after this call, SpECTRE also
     216             : defines a special `Exit` phase that is guaranteed to be executed after
     217             : all messages and entry methods have been processed.
     218             : 
     219             : > As described so far, the execution of individual Chares is
     220             : > “reactive”: When method A is invoked the chare executes this code,
     221             : > and so on. But very often, chares have specific life-cycles, and the
     222             : > sequence of entry methods they execute can be specified in a
     223             : > structured manner, while allowing for some localized non-determinism
     224             : > (e.g. a pair of methods may execute in any order, but when they both
     225             : > finish, the execution continues in a pre-determined manner, say
     226             : > executing a 3rd entry method).
     227             : 
     228             : Charm++ provides a special notation to simplify expression of such
     229             : control structures, but this requires writing specialized interface
     230             : files that are parsed by Charm++.  SpECTRE does not support this;
     231             : rather we split the executable into a set of user-defined phases. In
     232             : each phase, each parallel component will execute a user-defined list
     233             : of actions.
     234             : 
     235             : > The normal entry methods, being asynchronous, are not allowed to
     236             : > return any value, and are declared with a void return type.
     237             : 
     238             : SpECTRE's actions do not return any value to the calling component.
     239             : Instead when the action is finished it can call another action to send
     240             : data to an element of any parallel component.
     241             : 
     242             : > To support asynchronous method invocation and global object space,
     243             : > the RTS needs to be able to serialize (“marshall”) the parameters,
     244             : > and be able to generate global “names” for chares. For this purpose,
     245             : > programmers have to declare the chare classes and the signature of
     246             : > their entry methods in a special “.ci” file, called an interface
     247             : > file. Other than the interface file, the rest of a Charm++ program
     248             : > consists of just normal C++ code. The system generates several
     249             : > classes based on the declarations in the interface file, including
     250             : > “Proxy” classes for each chare class. Those familiar with various
     251             : > component models (such as CORBA) in the distributed computing world
     252             : > will recognize “proxy” to be a dummy, standin entity that refers to
     253             : > an actual entity. For each chare type, a “proxy” class exists. The
     254             : > methods of this “proxy” class correspond to the remote methods of
     255             : > the actual class, and act as “forwarders”. That is, when one invokes
     256             : > a method on a proxy to a remote object, the proxy marshalls the
     257             : > parameters into a message, puts adequate information about the
     258             : > target chare on the envelope of the message, and forwards it to the
     259             : > remote object. Individual chares, chare array, groups, node-groups,
     260             : > as well as the individual elements of these collections have a such
     261             : > a proxy. Multiple methods for obtaining such proxies are described
     262             : > in the manual. Proxies for each type of entity in Charm++ have some
     263             : > differences among the features they support, but the basic syntax
     264             : > and semantics remain the same - that of invoking methods on the
     265             : > remote object by invoking methods on proxies.
     266             : 
     267             : SpECTRE has wrapped all of this functionality in order to make it
     268             : easier to use.  SpECTRE automatically creates the interface files for
     269             : each parallel component using template metaprogramming.  SpECTRE
     270             : provides proxies for each parallel component that are all held in the
     271             : `GlobalCache` which is available to every parallel component.  In
     272             : order for actions to be called as entry methods on remote parallel
     273             : components, the arguments to the function call must be serializable.
     274             : Charm++ provides the `PUP` framework to serialize objects where `PUP`
     275             : stands for pack-unpack.  Since the PUP framework is used by Charm++
     276             : for checkpointing, load-balancing, and passing arguments when calling
     277             : actions, all user-defined classes with member data must define a `pup`
     278             : function.
     279             : 
     280             : > In terms of physical resources, we assume the parallel machine
     281             : > consists of one or more nodes, where a node is a largest unit over
     282             : > which cache coherent shared memory is feasible (and therefore, the
     283             : > maximal set of cores per which a single process can run. Each node
     284             : > may include one or more processor chips, with shared or private
     285             : > caches between them. Each chip may contain multiple cores, and each
     286             : > core may support multiple hardware threads (SMT for example).
     287             : > Charm++ recognizes two logical entities: a PE (processing element)
     288             : > and a logical node, or simply “node”. In a Charm++ program, a PE is
     289             : > a unit of mapping and scheduling: each PE has a scheduler with an
     290             : > associated pool of messages. Each chare is assumed to reside on one
     291             : > PE at a time. A logical node is implemented as an OS process. In
     292             : > non-SMP mode there is no distinction between a PE and a logical
     293             : > node. Otherwise, a PE takes the form of an OS thread, and a logical
     294             : > node may contain one or more PEs. Physical nodes may be partitioned
     295             : > into one or more logical nodes. Since PEs within a logical node
     296             : > share the same memory address space, the Charm++ runtime system
     297             : > optimizes communication between them by using shared
     298             : > memory. Depending on the runtime command-line parameters, a PE may
     299             : > optionally be associated with a subset of cores or hardware threads.
     300             : 
     301             : In other words, how Charm++ defines a node and a PE depends upon how
     302             : Charm++ was installed on a system.  The executable
     303             : `Executables/ParallelInfo` can be used to determine how many nodes and
     304             : PEs exist for a given Charm++ build and the runtime command-line
     305             : parameters passed when calling the executable.
     306             : 
     307             : For more details see the [Charm++
     308             : documentation](https://charm.readthedocs.io) and \ref
     309             : dev_guide_parallelization_foundations.
