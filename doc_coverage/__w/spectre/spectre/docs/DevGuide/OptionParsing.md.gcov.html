       5             : # Option Parsing {#dev_guide_option_parsing}
       6             : 
       7             : SpECTRE can read YAML configuration files at runtime to set parameters
       8             : and choose between classes implementing an interface.  %Options are
       9             : parsed during code initialization and can be used to construct objects
      10             : placed in the Parallel::GlobalCache and options passed to the
      11             : parallel components.  The types necessary to mark objects for parsing
      12             : are declared in `Options/Options.hpp`.
      13             : 
      14             : ## General option format
      15             : 
      16             : An option is defined by an "option tag", represented by a `struct`.  At minimum,
      17             : the struct must declare the type of the object to be parsed and provide a brief
      18             : description of the meaning.  The name of the option in the input file
      19             : defaults to the name of the struct (excluding any template parameters
      20             : and scope information), but can be overridden by providing a static
      21             : `name()` function.  Several other pieces of information, such as
      22             : suggestions, limits and grouping, may be provided if desired.  This
      23             : information is all included in the generated help output.
      24             : 
      25             : If an option has a suggested value, the value is specified in the
      26             : input file as usual, but a warning will be issued if the specified
      27             : value does not match the suggestion.
      28             : 
      29             : Examples:
      30             : \snippet Test_Options.cpp options_example_scalar_struct
      31             : \snippet Test_Options.cpp options_example_vector_struct
      32             : 
      33             : The option type can be any type understood natively by yaml-cpp
      34             : (fundamentals, `std::string`, and `std::map`, `std::vector`,
      35             : `std::list`, `std::array`, and `std::pair` of parsable types) and
      36             : types SpECTRE adds support for.  SpECTRE adds `std::unordered_map`
      37             : (but only with ordered keys), and various classes marked as
      38             : constructible in their declarations.
      39             : 
      40             : An option tag can be placed in a group by adding a `group` type alias to the
      41             : struct. The alias should refer to a type that, like option tags, defines a help
      42             : string and may override a static `name()` function.
      43             : 
      44             : Example:
      45             : \snippet Test_Options.cpp options_example_group
      46             : 
      47             : ## Constructible classes
      48             : 
      49             : A class that defines `static constexpr Options::String help` and a
      50             : typelist of option structs `options` can be created by the option
      51             : parser.  When the class is requested, the option parser will parse
      52             : each of the options in the `options` list, and then supply them to the
      53             : constructor of the class.  (See [Custom parsing](#custom-parsing)
      54             : below for more general creation mechanisms.)
      55             : 
      56             : Unlike option descriptions, which should be brief, the class help
      57             : string has no length limits and should give a description of the class
      58             : and any necessary discussion of its options beyond what can be
      59             : described in their individual help strings.
      60             : 
      61             : Creatable classes must be default constructible and move assignable.
      62             : 
      63             : The `Options::Context` is an optional argument to the constructor that should be
      64             : used when the constructor checks for validity of the input. If the input is
      65             : invalid, `PARSE_ERROR` is used to propagate the error message back through the
      66             : options ensuring that the error message will have a full backtrace so it is easy
      67             : for the user to diagnose.
      68             : 
      69             : Example:
      70             : \snippet Test_CustomTypeConstruction.cpp class_creation_example
      71             : 
      72             : Classes may use the Metavariables struct, which is effectively the compile time
      73             : input file, in their parsing by templating the `options` type alias or by taking
      74             : the Metavariables as a final argument to the constructor (after the
      75             : `Options::Context`).
      76             : 
      77             : Example:
      78             : \snippet Test_CustomTypeConstruction.cpp class_creation_example_with_metavariables
      79             : 
      80             : ## Factory
      81             : 
      82             : The factory interface creates an object of type
      83             : `std::unique_ptr<Base>` containing a pointer to some class derived
      84             : from `Base`.  The base class must define a type alias listing the
      85             : derived classes that can be created.
      86             : \code{cpp}
      87             :  using creatable_classes = tmpl::list<Derived1, ...>;
      88             : \endcode
      89             : The requested derived class is created in the same way as an
      90             : explicitly constructible class.
      91             : 
      92             : \anchor custom-parsing
      93             : ## Custom parsing
      94             : 
      95             : Occasionally, the requirements imposed by the default creation
      96             : mechanism are too stringent.  In these cases, the construction
      97             : algorithm can be overridden by providing a specialization of the
      98             : struct
      99             : \code{cpp}
     100             : template <typename T>
     101             : struct Options::create_from_yaml {
     102             :   template <typename Metavariables>
     103             :   static T create(const Options::Option& options);
     104             : };
     105             : \endcode
     106             : The create function can perform any operations required to construct
     107             : the object.
     108             : 
     109             : Example of using a specialization to parse an enum:
     110             : \snippet Test_CustomTypeConstruction.cpp enum_creation_example
     111             : 
     112             : Note that in the case where the `create` function does *not* need to use the
     113             : `Metavariables` it is recommended that a general implementation forward to an
     114             : explicit instantiation with `void` as the `Metavariables` type. The reason for
     115             : using `void` specialization is to reduce compile time. Since we only need one
     116             : full implementation of the function independent of what type `Metavariables` is,
     117             : we should only parse and compile it once. By having a specialization on `void`
     118             : (or some other non-metavariables type like `NoSuchType`) we can handle the
     119             : metavariables-independent case efficiently. As a concrete example, the general
     120             : definition and forward declaration of the `void` specialization in the header
     121             : file would be:

     123             : \snippet Test_CustomTypeConstruction.cpp enum_void_creation_header_example
     124             : 
     125             : while in the `cpp` file the definition of the `void` specialization is:
     126             :
