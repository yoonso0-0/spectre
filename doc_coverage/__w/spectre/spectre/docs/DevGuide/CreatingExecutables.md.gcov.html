       5             : # Creating Executables {#dev_guide_creating_executables}
       6             : 
       7             : There are several different types of executables that can be built:
       8             : - An executable that uses Charm++ to run in parallel,
       9             :   e.g. `Executables/ParallelInfo`
      10             : - An executable that does not use Charm++, does not run in parallel, and
      11             :   supplies its own `main`, e.g. `Executables/Benchmark`
      12             : - An executable that uses Charm++ to run in parallel but supplies its own `main`
      13             : - An executable that uses custom compilation or linking flags,
      14             :   e.g. `DebugPreprocessor`
      15             : - Executables used for evolutions or elliptic solves
      16             : 
      17             : ### Executable Using Charm++ for Parallelization
      18             : 
      19             : \ref tutorial_minimal_parallel_executable tutorial describes how to
      20             : add a new parallel executable.
      21             : 
      22             : Another simple example of an executable using Charm++ for
      23             : parallelization is in `src/Executables/Examples/HelloWorld`.  In this
      24             : example, the only additional phase (besides `Initialization` and
      25             : `Exit`) is `Execute`, and the phases are executed in order.
      26             : `SingletonHelloWorld` defines a single component `HelloWorld`
      27             : 
      28             : \snippet SingletonHelloWorld.hpp executable_example_singleton
      29             : 
      30             : which specifies via the `chare_type` type alias that it is a singleton
      31             : parallel component which means that only one such object will exist
      32             : across all processors used by the executable.  Each component must
      33             : define the static function `execute_next_phase` which is executed
      34             : during the phases (other than `Initialization` and `Exit`) defined in
      35             : the metavariables struct.  In `SingletonHelloWorld`, the
      36             : `PrintMessage` action is called during the `Execute` phase.
      37             : 
      38             : \snippet  SingletonHelloWorld.hpp executable_example_action
      39             : 
      40             : The `PrintMessage` action is executed on whatever process the singleton
      41             : component is created upon, and prints a message.
      42             : 
      43             : Executables can read in an input file (specified by the `--input-file`
      44             : argument) that will be parsed when the executable begins.  %Options
      45             : specified in the input file can be used to either place items in the
      46             : Parallel::GlobalCache (by specifying tags in the
      47             : `const_global_cache_tags` type alias of the metavariables, component
      48             : and action structs), to construct items in the db::DataBox of
      49             : components during initialization (by specifying tags in the
      50             : `initialization_tags` type alias of action struct), or be passed to
      51             : the `allocate_array` function of an array component (by specifying
      52             : tags in the `allocation_tags` type alias of the component).
      53             : `SingletonHelloWorld` specifies a single option
      54             : 
      55             : \snippet SingletonHelloWorld.hpp executable_example_options
      56             : 
      57             : which a string specifying a name that will be placed into the constant global
      58             : cache.  The string is fetched when performing the `PrintMessage` action. Items
      59             : in the constant global cache are stored once per node that the executable runs
      60             : on. An example input file for `SingletonHelloWorld` can be found in
      61             : `tests/InputFiles/ExampleExecutables/SingletonHelloWorld.yaml` and shows how to
      62             : specify the options (lines beginning with a `#` are comments and can be
      63             : ignored).
      64             : 
      65             : Furthermore among the included header files
      66             : 
      67             : \snippet  SingletonHelloWorld.hpp executable_example_includes
      68             : 
      69             : must be the appropriate header for each parallel component type, which in the
      70             : `SingletonHelloWorld` example is `AlgorithmSingleton.hpp`.  Note that
      71             : these headers are not in the source tree, but are generated automatically when
      72             : the code is compiled.
      73             : 
      74             : See [the Parallelization documentation](group__ParallelGroup.html#details)
      75             : for more details.
      76             : 
      77             : ### Executable Using Charm++ with Custom main()
      78             : 
      79             : While this is technically possible, it has not been tested. We recommend using
      80             : the Charm++ supplied main chare mechanism for the time being.
      81             : 
      82             : ### Executable Not Using Charm++
      83             : 
      84             : An example of an executable that does not use Charm++ for parallelization but
      85             : still can use all other infrastructure in SpECTRE is in
      86             : `src/Executables/Examples/HelloWorldNoCharm`. Adding a non-Charm++ executable to
      87             : SpECTRE mostly follows the standard way of adding an executable using CMake. The
      88             : only deviation is that the `CMakeLists.txt` file must tell Charm++ not to add a
      89             : `main()` by passing the link flags `-nomain-module -nomain`. This is done using
      90             : CMake's `set_target_properties`:
      91             : 
      92             : ```
      93             : set_target_properties(
      94             :   ${EXECUTABLE}
      95             :   PROPERTIES LINK_FLAGS "-nomain-module -nomain"
      96             :   )
      97             : ```
      98             : 
      99             : To add the executable as a target you must use the `add_spectre_executable`
     100             : function, which is a light weight wrapper around CMake's `add_executable`.
     101             : For example,
     102             : 
     103             : ```
     104             : add_spectre_executable(
     105             :   ${EXECUTABLE}
     106             :   EXCLUDE_FROM_ALL # Exclude from calls to `make` without a specified target
     107             :   HelloWorld.cpp
     108             :   )
     109             : ```
     110             : 
     111             : You can link in any of the SpECTRE libraries by adding them to the
     112             : `target_link_libraries`, for example:
     113             : 
     114             : ```
     115             : target_link_libraries(
     116             :   ${EXECUTABLE}
     117             :   DataStructures
     118             :   )
     119             : ```
     120             : 
     121             : We recommend that you add a test that the executable properly runs by adding an
     122             : input file to `tests/InputFiles` in an appropriate subdirectory. See
     123             : [`tests/InputFiles/ExampleExecutables/HelloWorldNoCharm.yaml`]
     124             : (https://github.com/sxs-collaboration/spectre/tree/develop/tests/InputFiles/
     125             : ExampleExecutables/HelloWorldNoCharm.yaml)
     126             : for an example.
     127             : The input file is passed to the executable using `--input-file
     128             : path/to/Input.yaml`. In the case of the executable not taking any input file
     129             : this is just used to generate a test that runs the executable.
     130             : 
     131             : For these types of executables `main` can take the usual `(int argc, char
     132             : *argv[])` and parse command line options. Executables not using Charm++ are just
     133             : standard executables that can link in any of the libraries in SpECTRE.
     134             : 
     135             : \warning
     136             : Currently calling `Parallel::abort` results in a segfault deep inside Charm++
     137             : code. However, the error messages from `ASSERT` and `ERROR` are still printed.
     138             : 
     139             : ### Executable With Custom Compilation or Linking Flags
     140             : 
     141             : Use the CMake function `set_target_properties` to add flags to an executable. To
     142             : call a completely custom compiler invocation you should use the
     143             : `add_custom_target` CMake function. The need for the `custom_target` level of
     144             : control is rare and should generally be avoided since it adds quite a bit of
     145             : technical debt to the code base. Thus, it is not explained here. If you are
     146             : certain you need it you can see the `DebugPreprocessor` executable's
     147             : `CMakeLists.txt` file for an example.
     148             : 
     149             : ### Executable Used for Evolution or Elliptic Solve
     150             : 
     151             : Once they are written, see the tutorials specific to evolution and elliptic
     152             : solves.
