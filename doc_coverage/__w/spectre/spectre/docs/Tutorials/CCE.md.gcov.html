       5             : # %Running CCE {#tutorial_cce}
       6             : 
       7             : The basic instructions for getting up and running with a stand-alone
       8             : CCE using external data are:
       9             : - Clone spectre and build the CharacteristicExtract target
      10             : - *Note*: CCE currently requires the linked HDF5 library to be built
      11             :   thread-safe. As this is a significant restriction on many systems,
      12             :   we hope to eliminate this requirement in the future.
      13             : - At this point (provided the build succeeds) you should have the
      14             :   executable `bin/CharacteristicExtract` in the build directory; You can now
      15             :   run it using an input file to provide options. The input file
      16             :   `tests/InputFiles/Cce/CharacteristicExtract.yaml` from the spectre
      17             :   source tree can help you get started with writing your input file.
      18             :   There are a few important notes there:
      19             :   - For resolution, the example input file has lmax (`Cce.LMax`) of 12, and
      20             :     filter lmax (`Filtering.FilterLMax`) of 10; that'll run pretty fast but
      21             :     might be a little low for full precision. lmax 16, filter 14 should be
      22             :     pretty good, and typically precision doesn't improve above lmax 24,
      23             :     filter 22 (be sure to update the filter as you update lmax -- it should
      24             :     generally be around 2 lower than the maximum l to reduce possible aliasing).
      25             :   - If you want to just run through the end of the provided worldtube data,
      26             :     you can just omit the `EndTime` option and the executable will figure it
      27             :     out from the worldtube file.
      28             :   - The `ScriOutputDensity` adds extra interpolation points to the output,
      29             :     which is useful for finite-difference derivatives on the output data, but
      30             :     otherwise it'll just unnecessarily inflate the output files, so if you
      31             :     don't need the extra points, best just set it to 1.
      32             :   - If you're extracting at 100M or less, best to reduce the `TargetStepSize`,
      33             :     to around .5 at 100M and lower yet for nearer extraction.
      34             :   - The `InitializeJ` in the example file uses `InverseCubic` which is a pretty
      35             :     primitive scheme, but early tests indicate that it gives the best results
      36             :     for most systems.
      37             :     If initial data is a concern, you can also try replacing the `InverseCubic`
      38             :     entry with:
      39             :   ```
      40             :     NoIncomingRadiation:
      41             :       AngularCoordTolerance: 1.0e-13
      42             :       MaxIterations: 500
      43             :       RequireConvergence: true
      44             :   ```
      45             :   which are probably pretty good choices for those parameters,
      46             :   and the `RequireConvergence: true` will cause the iterative solve in
      47             :   this version to error out if it doesn't find a good frame.
      48             : 
      49             : - An example of an appropriate submission command for slurm systems is:
      50             :   ```
      51             :   srun -n 1 -c 1 path/to/build/bin/CharacteristicExtract ++ppn 3 \
      52             :  --input-file path/to/input.yaml
      53             :   ```
      54             :   CCE doesn't currently scale to more than 4 cores, so those slurm options are
      55             :   best.
      56             : - CCE will work faster if the input worldtube hdf5 file is chunked in small
      57             :   numbers of complete rows.
      58             :   This is relevant because by default, SpEC writes its worldtube files
      59             :   chunked along full time-series columns, which is efficient for
      60             :   compression, but not for reading in to SpECTRE -- in that case,
      61             :   it is recommended to rechunk the input file before running CCE
      62             :   for maximum performance. This can be done, for instance, using h5py
      63             :   (you will need to fill in filenames appropriate to your case in place
      64             :   of "BondiCceR0050.h5" and "RechunkBondiCceR0050.h5"):
      65             :   ```py
      66             :   import h5py
      67             :   input_file = "BondiCceR0050.h5"
      68             :   output_file = "RechunkBondiCceR0050.h5"
      69             :   with h5py.File(input_file,'r') as input_h5,\
      70             :     h5py.File(output_file, 'w') as output_h5:
      71             :     for dset in input_h5:
      72             :         if("Version" in dset):
      73             :             output_h5[dset] = input_h5[dset][()]
      74             :             continue
      75             :         number_of_columns = input_h5[dset][()].shape[1]
      76             :         output_h5.create_dataset(dset, data=input_h5[dset],
      77             :                                  maxshape=(None, number_of_columns),
      78             :                                  chunks=(4, number_of_columns), dtype='d')
      79             :         for attribute in input_h5[dset].attrs.keys():
      80             :             output_h5[dset].attrs[attribute] = input_h5[dset].attrs[attribute]
      81             :   ```
      82             : - The output data will be written as spin-weighted spherical harmonic
      83             :   modes, one physical quantity per dataset, and each row will
      84             :   have the time value followed by the real and imaginary parts
      85             :   of the complex modes in m-varies-fastest order.
      86             : 
      87             : Once you have the volume data output file from a successful CCE run, you can
      88             : confirm the integrity of the h5 file and its contents by running
      89             : ```
      90             : h5ls CharacteristicExtractVolumeData0.h5
      91             : ```
      92             : 
      93             : For the volume file produced by a successful run, the output of the `h5ls`
      94             : should resemble
      95             : ```
      96             : EthInertialRetardedTime.dat Dataset {3995/Inf, 163}
      97             : News.dat                 Dataset {3995/Inf, 163}
      98             : Psi0.dat                 Dataset {3995/Inf, 163}
      99             : Psi1.dat                 Dataset {3995/Inf, 163}
     100             : Psi2.dat                 Dataset {3995/Inf, 163}
     101             : Psi3.dat                 Dataset {3995/Inf, 163}
     102             : Psi4.dat                 Dataset {3995/Inf, 163}
     103             : Strain.dat               Dataset {3995/Inf, 163}
     104             : src.tar.gz               Dataset {3750199}
     105             : ```
     106             : 
     107             : The `Strain.dat` represents the asymptotic transverse-traceless contribution
     108             : to the metric scaled by the Bondi radius (to give the asymptotically leading
     109             : part), the `News.dat` represents the first derivative of the strain, and each
     110             : of the `Psi...` datasets represent the Weyl scalars, each scaled by the
     111             : appropriate factor of the Bondi-Sachs radius to retrieve the asymptotically
     112             : leading contribution.
     113             : 
     114             : The `EthInertialRetardedTime.dat` is a diagnostic dataset that represents the
     115             : angular derivative of the inertial retarded time, which determines the
     116             : coordinate transformation that is performed at scri+.
     117             : 
     118             : The following python script will load data from a successful CCE run and
     119             : construct a plot of all of the physical waveform data.
     120             : ```py
     121             : import matplotlib.pyplot as plt
     122             : import numpy as np
     123             : import h5py as h5
     124             : 
     125             : 
     126             : def spectre_real_mode_index(l, m):
     127             :     return 2 * (l**2 + l + m)
     128             : 
     129             : 
     130             : def spectre_imag_mode_index(l, m):
     131             :     return 2 * (l**2 + l + m) + 1
     132             : 
     133             : 
     134             : def get_modes_from_block_output(filename, dataset, modes=[[2, 2], [3, 3]]):
     135             :     with h5.File(filename, "r") as h5_file:
     136             :         timeseries_data = (h5_file[dataset + ".dat"][()][:, [0] + list(
     137             :             np.array([[
     138             :                 spectre_real_mode_index(x[0], x[1]),
     139             :                 spectre_imag_mode_index(x[0], x[1])
     140             :             ] for x in modes]).flatten())])
     141             :     return timeseries_data
     142             : 
     143             : 
     144             : plot_quantities = ["Strain", "News", "Psi0", "Psi1", "Psi2", "Psi3", "Psi4"]
     145             : mode_set = [[2, 2], [3, 3]]
     146             : filename = "CharacteristicExtractVolume0.h5"
     147             : output_plot_filename = "CCE_plot.pdf"
     148             : 
     149             : legend = []
     150             : for (mode_l, mode_m) in mode_set:
     151             :     legend = np.append(legend, [
     152             :         r"Re $Y_{" + str(mode_l) + r"\," + str(mode_m) + r"}$",
     153             :         r"Im $Y_{" + str(mode_l) + r"\," + str(mode_m) + r"}$"
     154             :     ])
     155             : 
     156             : plt.figure(figsize=(8, 1.9 * len(plot_quantities)))
     157             : for i in range(len(plot_quantities)):
     158             :     ax = plt.subplot(len(plot_quantities), 1, i + 1)
     159             :     timeseries = np.transpose(
     160             :         get_modes_from_block_output(filename, plot_quantities[i], mode_set))
     161             :     for j in range(len(mode_set)):
     162             :         plt.plot(timeseries[0], timeseries[j + 1], linestyle='--', marker='')
     163             :     ax.set_xlabel("Simulation time (M)")
     164             :     ax.set_ylabel("Mode coefficient")
     165             :     ax.set_title(plot_quantities[i])
     166             : plt.tight_layout()
     167             : plt.savefig(output_plot_filename, dpi=400)
     168             : plt.clf()
     169             : ```
     170             : 
     171             : ### Input data formats
     172             : 
     173             : The worldtube data must be constructed as spheres of constant coordinate
     174             : radius, and (for the time being) written to a filename of the format
     175             : `...CceRXXXX.h5`, where the `XXXX` is to be replaced by the integer for which
     176             : the extraction radius is equal to `XXXX`M. For instance, a 100M extraction
     177             : should have filename `...CceR0100.h5`. This scheme of labeling files with the
     178             : extraction radius is constructed for compatibility with SpEC worldtube data.
     179             : 
     180             : There are two possible formats of the input data, one based on the Cauchy metric
     181             : at finite radius, and one based on Bondi data. The metric data format must be
     182             : provided as spherical harmonic modes with the following datasets:
     183             : - `gxx.dat`, `gxy.dat`, `gxz.dat`, `gyy.dat`, `gyz.dat`, `gzz.dat`
     184             : - `Drgxx.dat`, `Drgxy.dat`, `Drgxz.dat`, `Drgyy.dat`, `Drgyz.dat`, `Drgzz.dat`
     185             : - `Dtgxx.dat`, `Dtgxy.dat`, `Dtgxz.dat`, `Dtgyy.dat`, `Dtgyz.dat`, `Dtgzz.dat`
     186             : - `Shiftx.dat`, `Shifty.dat`, `Shiftz.dat`
     187             : - `DrShiftx.dat`, `DrShifty.dat`, `DrShiftz.dat`
     188             : - `DtShiftx.dat`, `DtShifty.dat`, `DtShiftz.dat`
     189             : - `Lapse.dat`
     190             : - `DrLapse.dat`
     191             : - `DtLapse.dat`
     192             : 
     193             : In this format, each row must start with the time stamp, and the remaining
     194             : values are the complex modes in m-varies-fastest format.
     195             : 
     196             : The second format is Bondi-Sachs metric component data.
     197             : This format is far more space-efficient (by around a factor of 4), and SpECTRE
     198             : provides a separate executable for converting to the Bondi-Sachs worldtube
     199             : format, `ReduceCceWorldtube`.
     200             : The `ReduceCceWorldtube` executable should be run on a Cauchy metric worldtube
     201             : file, and will produce a corresponding 'reduced' Bondi-Sachs worldtube file.
     202             : The basic command-line arguments for the executable are:
     203             : ```
     204             : ReduceCceWorldtube --input-file CceR0050.h5 --output-file BondiCceR0050.h5\
     205             :  --lmax_factor 3
     206             : ```
     207             : The argument `--lmax_factor` determines the factor by which the resolution at
     208             : which the boundary computation that is run will exceed the resolution of the
     209             : input and output files.
     210             : Empirically, we have found that `lmax_factor` of 3 is sufficient to achieve
     211             : roundoff precision in all boundary data we have attempted, and an `lmax_factor`
     212             : of 2 is usually sufficient to vastly exceed the precision of the simulation that
     213             : provided the boundary dataset.
     214             : 
     215             : The format is similar to the metric components, except in spin-weighted
     216             : spherical harmonic modes, and the real (spin-weight-0) quantities omit the
     217             : redundant negative-m modes and imaginary parts of m=0 modes.
     218             : The quantities that must be provided by the Bondi-Sachs metric data format are:
     219             : - `Beta.dat`
     220             : - `DrJ.dat`
     221             : - `DuR.dat`
     222             : - `H.dat`
     223             : - `J.dat`
     224             : - `Q.dat`
     225             : - `R.dat`
     226             : - `U.dat`
     227             : - `W.dat`
     228             : 
     229             : The Bondi-Sachs data may also be used directly for CCE input data.
     230             : To specify that the input type is in 'reduced' Bondi-Sachs form, use:
     231             : ```
     232             : ...
     233             : Cce:
     234             :   H5IsBondiData: True
     235             : ...
     236             : ```
     237             : Otherwise, for the Cauchy metric data format, use:
     238             : ```
     239             : ...
     240             : Cce:
     241             :   H5IsBondiData: False
     242             : ...
     243             : ```
