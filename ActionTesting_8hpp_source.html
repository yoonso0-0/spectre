40 /*!
41  * \ingroup TestingFrameworkGroup
42  * \brief Structures used for mocking the parallel components framework in order
43  * to test actions.
44  *
45  * The ActionTesting framework is designed to mock the parallel components so
46  * that actions and sequences of actions can be tested in a controlled
47  * environment that behaves effectively identical to the actual parallel
48  * environment.
49  *
50  * ### The basics
51  *
52  * The action testing framework (ATF) works essentially identically to the
53  * parallel infrastructure. A metavariables must be supplied which must at least
54  * list the components used (`using component_list = tmpl::list<>`) and the
55  * phases (`enum class Phase {}`). As a simple example, let's look at the test
56  * for the `Parallel::Actions::TerminatePhase` action.
57  *
58  * The `Metavariables` is given by:
59  *
60  * \snippet Test_TerminatePhase.cpp metavariables
61  *
62  * The component list in this case just contains a single component that is
63  * described below. The `Phase` enum contains the standard `Initialization` and
64  * `Exit` phases. A `Testing` phase is also added, but the name of the phase has
65  * no special meaning. Multiple phases with different names could instead be
66  * used.
67  *
68  * The component is templated on the metavariables, which, while not always
69  * necessary, eliminates some compilation issues that may arise otherwise. The
70  * component for the `TerminatePhase` test is given by:
71  *
72  * \snippet Test_TerminatePhase.cpp component
73  *
74  * Just like with the standard parallel code, a `metavariables` type alias must
75  * be present. The chare type is always `ActionTesting::MockArrayChare` since
76  * singletons may be treated during action testing as just a one-element array
77  * component. The `index_type` must be whatever the actions will use to index
78  * the array. In the case of a singleton `int` is recommended. Finally, the
79  * `phase_dependent_action_list` must be specified just as in the cases for
80  * parallel executables. In this case only the `Testing` phase is used and only
81  * has the `TerminatePhase` action listed.
82  *
83  * The `SPECTRE_TEST_CASE` is
84  *
85  * \snippet Test_TerminatePhase.cpp test case
86  *
87  * The type alias `component = Component<Metavariables>` is just used to reduce
88  * the amount of typing needed. The ATF provides a
89  * `ActionTesting::MockRuntimeSystem` class, which is the code that takes the
90  * place of the parallel runtime system (RTS) (e.g. Charm++) that manages the
91  * actions and components. Components are added to the RTS using the
92  * `ActionTesting::emplace_component()` function, which takes the runner by
93  * `not_null`, the array index of the component being inserted (`0` in the above
94  * example, but this is arbitrary), and optionally a parameter pack of
95  * additional arguments that are forwarded to the constructor of the component.
96  * These additional arguments are how input options (set in the
97  * `initialization_tags` type alias of the parallel component) get passed to
98  * parallel components.
99  *
100  * With the ATF, everything is controlled explicitly, providing the ability to
101  * perform full introspection into the state of the RTS at virtually any point
102  * during the execution. This means that the phase is not automatically advanced
103  * like it is in a parallel executable, but instead must be advanced by calling
104  * the `ActionTesting::set_phase` function. While testing the `TerminatePhase`
105  * action we are only interesting in checking whether the algorithm has set the
106  * `terminate` flag. This is done using the `ActionTesting::get_terminate()`
107  * function and is done for each distributed object (i.e. per component). The
108  * next action in the action list for the current phase is invoked by using the
109  * `ActionTesting::next_action()` function, which takes as arguments the runner
110  * by `not_null` and the array index of the distributed object to invoke the
111  * next action on.
112  *
113  * ### InitializeDataBox and action introspection
114  *
115  * Having covered the very basics of ATF let us look at the test for the
116  * `Actions::Goto` action. We will introduce the functionality
117  * `InitializeDataBox`,`force_next_action_to_be`, `get_next_action_index`, and
118  * `get_databox_tag`. The `Goto<the_label_class>` action changes the next action
119  * in the algorithm to be the `Actions::Label<the_label_class>`. For this test
120  * the Metavariables is:
121  *
122  * \snippet Test_Goto.cpp metavariables
123  *
124  * You can see that this time there are two test phases, `TestGoto` and
125  * `TestRepeatUntil`.
126  *
127  * The component for this case is quite a bit more complicated so we will go
128  * through it in detail. It is given by
129  *
130  * \snippet Test_Goto.cpp component
131  *
132  * Just as before there are `metavariables`, `chare_type`, and `array_index`
133  * type aliases. The `repeat_until_phase_action_list` is the list of iterable
134  * actions that are called during the `TestRepeatUntil` phase. The
135  * `Initialization` phase action list now has the action
136  * `ActionTesting::InitializeDataBox`, which takes simple tags and compute tags
137  * that will be added to the DataBox in the initial phase. How the values for
138  * the simple tags are set will be made clear below when we discuss the
139  * `ActionTesting::emplace_component_and_initialize()` function. The action
140  * lists for the `TestGoto` and `TestRepeatUntil` phases are fairly simple and
141  * not the focus of this discussion.
142  *
143  * Having discussed the metavariables and component, let us now look at the test
144  * case.
145  *
146  * \snippet Test_Goto.cpp test case
147  *
148  * Just as for the `TerminatePhase` test we have some type aliases to reduce the
149  * amount of typing needed and to make the test easier to read. The runner is
150  * again created with the default constructor. However, the component is now
151  * inserted using the `ActionTesting::emplace_component_and_initialize()`
152  * function. The third argument to
153  * `ActionTesting::emplace_component_and_initialize()` is a
154  * `tuples::TaggedTuple` of the simple tags, which is to be populated with the
155  * initial values for the component. Note that there is no call to
156  * `ActionTesting::set_phase (&runner, Metavariables::Phase::Initialization)`:
157  * this and the required action invocation is handled internally by
158  * `ActionTesting::emplace_component_and_initialize()`.
159  *
160  * Once the phase is set the next action to be executed is set to be
161  * `Actions::Label<Label1>` by calling the
162  * `MockRuntimeSystem::force_next_action_to_be()` member function of the
163  * runner. The argument to the function is the array index of the component for
164  * which to set the next action. After the `Label<Label1>` action is invoked we
165  * check that the next action is the fourth (index `3` because we use zero-based
166  * indexing) by calling the `MockRuntimeSystem::get_next_action_index()` member
167  * function. For clarity, the indices of the actions in the `Phase::TestGoto`
168  * phase are:
169  * - 0: `Actions::Goto<Label1>`
170  * - 1: `Actions::Label<Label2>`
171  * - 2: `Actions::Label<Label1>`
172  * - 3: `Actions::Goto<Label2>`
173  *
174  * ### DataBox introspection
175  *
176  * Since the exact DataBox type of any component at some point in the action
177  * list may not be known, the `ActionTesting::get_databox_tag()` function is
178  * provided. An example usage of this function can be seen in the last line of
179  * the test snippet above. The component and tag are passed as template
180  * parameters while the runner and array index are passed as arguments. It is
181  * also possible to retrieve DataBox directly using the
182  * `ActionTesting::get_databox()` function if the DataBox type is known:
183  *
184  * \snippet Test_ActionTesting.cpp get databox
185  *
186  * There is also a `gsl::not_null` overload of `ActionTesting::get_databox()`
187  * that can be used to mutate tags in the DataBox. It is also possible to check
188  * if an item can be retrieved from the DataBox using the
189  * `ActionTesting::tag_is_retrievable()` function as follows:
190  *
191  * \snippet Test_ActionTesting.cpp tag is retrievable
192  *
193  * ### Stub actions and invoking simple and threaded actions
194  *
195  * A simple action can be invoked on a distributed object or component using
196  * the `ActionTesting::simple_action()` function:
197  *
198  * \snippet Test_ActionTesting.cpp invoke simple action
199  *
200  * A threaded action can be invoked on a distributed object or component using
201  * the `ActionTesting::threaded_action()` function:
202  *
203  * \snippet Test_ActionTesting.cpp invoke threaded action
204  *
205  * Sometimes an individual action calls other actions but we still want to be
206  * able to test the action and its effects in isolation. To this end the ATF
207  * supports replacing calls to actions with calls to other actions. For example,
208  * instead of calling `simple_action_a` we want to call `simple_action_a_mock`
209  * which just verifies the data received is correct and sets a flag in the
210  * DataBox that it was invoked. This can be done by setting the type aliases
211  * `replace_these_simple_actions` and `with_these_simple_actions` in the
212  * parallel component definition as follows:
213  *
214  * \snippet Test_ActionTesting.cpp simple action replace
215  *
216  * The length of the two type lists must be the same and the Nth action in
217  * `replace_these_simple_actions` gets replaced by the Nth action in
218  * `with_these_simple_actions`. Note that `simple_action_a_mock` will get
219  * invoked in any context that `simple_action_a` is called to be invoked. The
220  * same feature also exists for threaded actions:
221  *
222  * \snippet Test_ActionTesting.cpp threaded action replace
223  *
224  * Furthermore, simple actions invoked from an action are not run immediately.
225  * Instead, they are queued so that order randomization and introspection may
226  * occur. The simplest form of introspection is checking whether the simple
227  * action queue is empty:
228  *
229  * \snippet Test_ActionTesting.cpp simple action queue empty
230  *
231  * The `ActionTesting::invoke_queued_simple_action()` invokes the next queued
232  * simple action:
233  *
234  * \snippet Test_ActionTesting.cpp invoke queued simple action
235  *
236  * Note that the same functionality exists for threaded actions. The functions
237  * are `ActionTesting::is_threaded_action_queue_empty()`, and
238  * `ActionTesting::invoke_queued_threaded_action()`.
239  *
240  * ### Mocking or replacing components with stubs
241  *
242  * An action can invoke an action on another parallel component. In this case we
243  * need to be able to tell the mocking framework to replace the component the
244  * action is trying to invoke the other action on and instead use a different
245  * component that we have set up to mock the original component. For example,
246  * the action below invokes an action on `ComponentB`
247  *
248  * \snippet Test_ActionTesting.cpp action call component b
249  *
250  * Let us assume we cannot use the component `ComponentB` in our test and that
251  * we need to mock it. We do so using the type alias `component_being_mocked` in
252  * the mock component:
253  *
254  * \snippet Test_ActionTesting.cpp mock component b
255  *
256  * When creating the runner `ComponentBMock` is emplaced:
257  *
258  * \snippet Test_ActionTesting.cpp initialize component b
259  *
260  * Any checks and function calls into the ATF also use `ComponentBMock`. For
261  * example:
262  *
263  * \snippet Test_ActionTesting.cpp component b mock checks
264  *
265  * ### Const global cache tags
266  *
267  * Actions sometimes need tags/items to be placed into the
268  * `Parallel::GlobalCache`. Once the list of tags for the global cache has
269  * been assembled, the associated objects need to be inserted. This is done
270  * using the constructor of the `ActionTesting::MockRuntimeSystem`. For example,
271  * consider the tags:
272  *
273  * \snippet Test_ActionTesting.cpp tags for const global cache
274  *
275  * These are added into the global cache by, for example, the metavariables:
276  *
277  * \snippet Test_ActionTesting.cpp const global cache metavars
278  *
279  * A constructor of `ActionTesting::MockRuntimeSystem` takes a
280  * `tuples::TaggedTuple` of the const global cache tags. If you know the order
281  * of the tags in the `tuples::TaggedTuple` you can use the constructor without
282  * explicitly specifying the type of the `tuples::TaggedTuple` as follows:
283  *
284  * \snippet Test_ActionTesting.cpp constructor const global cache tags known
285  *
286  * If you do not know the order of the tags but know all the tags that are
287  * present you can use another constructor where you explicitly specify the
288  * `tuples::TaggedTuple` type:
289  *
290  * \snippet Test_ActionTesting.cpp constructor const global cache tags unknown
291  *
292  * ### Inbox tags introspection
293  *
294  * The inbox tags can also be retrieved from a component by using the
295  * `ActionTesting::get_inbox_tag` function. Both a const version:
296  *
297  * \snippet Test_ActionTesting.cpp const get inbox tags
298  *
299  * and a non-const version exist:
300  *
301  * \snippet Test_ActionTesting.cpp get inbox tags
302  *
303  * The non-const version can be used like in the above example to clear or
304  * otherwise manipulate the inbox tags.
305  *
306  * ###

312 // Initializes the DataBox values not set via the GlobalCache. This is
313 // done as part of an `Initialization` phase and is triggered using the
314 // `emplace_component_and_initialize` function.
315 template <typename... SimpleTags, typename ComputeTagsList>
316 struct InitializeDataBox<tmpl::list<SimpleTags...>, ComputeTagsList> {
317  // these type aliases must be different from those used by `SetupDataBox` to
318  // avoid a clash.
319  using action_testing_simple_tags = tmpl::list<SimpleTags...>;
320  using action_testing_compute_tags = ComputeTagsList;
321  using InitialValues = tuples::TaggedTuple<SimpleTags...>;
322 
323  template <typename DbTagsList, typename... InboxTags, typename Metavariables,
324  typename ActionList, typename ParallelComponent,
325  typename ArrayIndex,
326  Requires<not tmpl2::flat_any_v<
327  tmpl::list_contains_v<DbTagsList, SimpleTags>...>> = nullptr>
328  static auto apply(db::DataBox<DbTagsList>& box,
329  const tuples::TaggedTuple<InboxTags...>& /*inboxes*/,
330  const Parallel::GlobalCache<Metavariables>& /*cache*/,
331  const ArrayIndex& /*array_index*/,
332  const ActionList /*meta*/,
333  const ParallelComponent* const /*meta*/) noexcept {
334  if (not initial_values_valid_) {
335  ERROR(
336  "The values being used to construct the initial DataBox have not "
337  "been set.");
338  }
339  initial_values_valid_ = false;
340  return std::make_tuple(
341  db::create_from<db::RemoveTags<>, db::AddSimpleTags<SimpleTags...>,
342  ComputeTagsList>(
343  std::move(box),
344  std::move(tuples::get<SimpleTags>(initial_values_))...));
345  }
346 
347  template <typename DbTagsList, typename... InboxTags, typename Metavariables,
348  typename ActionList, typename ParallelComponent,
349  typename ArrayIndex,
350  Requires<tmpl2::flat_any_v<
351  tmpl::list_contains_v<DbTagsList, SimpleTags>...>> = nullptr>
352  static std::tuple<db::DataBox<DbTagsList>&&> apply(
353  db::DataBox<DbTagsList>& /*box*/,
354  const tuples::TaggedTuple<InboxTags...>& /*inboxes*/,
355  const Parallel::GlobalCache<Metavariables>& /*cache*/,
356  const ArrayIndex& /*array_index*/, const ActionList /*meta*/,
357  const ParallelComponent* const /*meta*/) noexcept {
358  ERROR(
359  "Tried to apply ActionTesting::InitializeDataBox even though one or "
360  "more of its tags are already in the DataBox. Did you call next_action "
361  "too many times in the initialization phase?");
362  }
